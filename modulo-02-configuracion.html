2.1 — CLAUDE.md

Archivo markdown que le da a Claude instrucciones persistentes para tu proyecto. Se carga en el contexto al inicio de cada sesion. No es configuracion forzada — es contexto, asi que la forma en que escribes las instrucciones afecta que tan bien Claude las sigue.

Donde colocarlo (multiples niveles)

Scope Ubicacion Compartido? Proposito
Managed /Library/Application Support/ClaudeCode/CLAUDE.md (macOS) Toda la org Politicas enterprise, estandares corporativos
Proyecto ./CLAUDE.md o ./.claude/CLAUDE.md Equipo (git) Convenciones del proyecto, comandos de build/test
Usuario ~/.claude/CLAUDE.md Solo tu Preferencias personales en todos los proyectos
Subdirectorio ./cualquier/sub/CLAUDE.md Equipo (git) Instrucciones especificas de esa carpeta

Herencia y orden de carga

  • Claude camina hacia arriba en el arbol de directorios cargando cada CLAUDE.md que encuentra
  • Si ejecutas Claude en foo/bar/, carga foo/bar/CLAUDE.md Y foo/CLAUDE.md
  • Los de subdirectorios se cargan bajo demanda cuando Claude lee archivos ahi (no al inicio)
  • Reglas de usuario (~/.claude/rules/) cargan antes que las del proyecto — proyecto tiene mayor prioridad
  • Managed CLAUDE.md no se puede excluir nunca

Imports con @

Puedes referenciar otros archivos directamente dentro de CLAUDE.md:

# Instrucciones del Proyecto

Ver @README para el overview del proyecto.
Ver @package.json para los comandos npm disponibles.

# Git workflow
Seguir las instrucciones en @docs/git-instructions.md

# Importar instrucciones personales desde home
@~/.claude/mis-instrucciones-proyecto.md
  • Rutas relativas (se resuelven desde el archivo que importa) o absolutas
  • Imports recursivos hasta 5 niveles de profundidad
  • El primer import externo dispara un dialogo de aprobacion
ⓘ Compatibilidad con AGENTS.md

Si tu repo ya tiene AGENTS.md para otras herramientas, importalo desde CLAUDE.md:

@AGENTS.md

## Claude Code
Usar plan mode para cambios en `src/billing/`.

.claude/rules/ — Instrucciones modulares

Para proyectos grandes, divide instrucciones en archivos separados:

.claude/
  rules/
    code-style.md      # Siempre se carga
    testing.md          # Siempre se carga
    api-rules.md        # Solo cuando trabaja en src/api/ (con frontmatter paths)

Las reglas pueden ser condicionales por path usando frontmatter YAML:

---
paths:
  - "src/api/**/*.ts"
---

# Reglas para el API
- Todos los endpoints deben validar input
- Usar formato estandar de error response
- Incluir tests de integracion para cada endpoint

Globs soportados: **/*.ts, src/**/*, *.md, **/*.{ts,tsx}

Generar CLAUDE.md automaticamente

# Genera un CLAUDE.md analizando tu codebase
/init

# Version interactiva multi-fase
export CLAUDE_CODE_NEW_INIT=true
/init
✔ Buena practica — CLAUDE.md efectivo
# CLAUDE.md - Proyecto API

## Build y Test
- Build: `npm run build`
- Test: `npm test`
- Lint: `npm run lint`
- Test individual: `npm test -- --grep "nombre"`

## Convenciones
- Indentacion: 2 espacios
- Nombres de archivos: kebab-case
- Exports: named exports, nunca default
- Errores API: usar `AppError` de `src/lib/errors.ts`

## Arquitectura
- Handlers en `src/api/handlers/`
- Servicios en `src/services/`
- Cada handler tiene su test en `__tests__/`

Especifico, verificable, corto (~30 lineas). Claude puede seguir cada instruccion sin ambiguedad.

✘ Mala practica — CLAUDE.md inefectivo
# CLAUDE.md

Este es un proyecto de Node.js que usa TypeScript.
Por favor formatea el codigo correctamente y sigue
las mejores practicas. Asegurate de que el codigo
sea limpio y mantenible. Los tests son importantes.
Trata de no romper nada. El proyecto es complejo
asi que ten cuidado. Usamos una base de datos
PostgreSQL y Redis para cache. El frontend esta
en React. Tenemos muchos microservicios...

(... 500 lineas mas de texto vago ...)

Vago, no verificable, demasiado largo. Claude no puede seguir "formatea correctamente" ni "ten cuidado" porque no son acciones concretas. Consume contexto innecesariamente.

claudeMdExcludes

En monorepos, excluye CLAUDE.md irrelevantes en settings.local.json:

{
  "claudeMdExcludes": [
    "**/otro-equipo/CLAUDE.md",
    "/home/user/monorepo/legacy/.claude/rules/**"
  ]
}
⚠ Tamano maximo recomendado

Maximo ~200 lineas por archivo CLAUDE.md. Archivos mas largos consumen contexto valioso y reducen la adherencia a las instrucciones. Si necesitas mas, usa .claude/rules/ para modularizar.

ⓘ Tip: comentarios HTML

Los comentarios HTML (<!-- nota -->) se eliminan antes de inyectar al contexto. Usalos para notas humanas que ahorran tokens de contexto.

✔ Buena practica — Separar shared vs personal
// .claude/settings.json (compartido con el equipo)
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": ["Bash(npm test *)", "Bash(npm run lint)"],
    "deny": ["Bash(git push *)", "Read(./.env*)"]
  },
  "hooks": {
    "PostToolUse": [{ "matcher": "Edit", "command": "npm run lint --fix $FILE" }]
  }
}

// .claude/settings.local.json (solo tu)
{
  "model": "opus",
  "effortLevel": "high",
  "claudeMdExcludes": ["**/team-mobile/CLAUDE.md"]
}

El equipo comparte permisos y hooks. Tu personalizas modelo y excluyes CLAUDE.md de equipos que no te afectan.

✘ Mala practica — Mezclar todo en un solo archivo
// .claude/settings.json (committeado a git)
{
  "model": "opus",           // Forzas TU modelo al equipo
  "effortLevel": "max",     // Gasto innecesario para otros
  "claudeMdExcludes": [      // TUS exclusiones afectan a todos
    "**/docs/CLAUDE.md"
  ],
  "permissions": {
    "allow": ["Bash(rm -rf *)"] // PELIGROSO: permite borrado masivo a todos
  }
}

Preferencias personales en el archivo compartido afectan a todo el equipo. Permisos peligrosos quedan committeados.

2.3 — .claudeignore (no existe nativamente)

No existe un archivo .claudeignore nativo en Claude Code. La forma oficial de excluir archivos es con reglas deny en settings.json.

Como lograr el mismo efecto

// En .claude/settings.json o settings.local.json
{
  "permissions": {
    "deny": [
      "Read(./.env)",            // Archivo exacto
      "Read(./.env.*)",           // Wildcard de extension
      "Read(./secrets/**)",       // Directorio recursivo
      "Read(./node_modules/**)",  // Dependencias
      "Read(./build/**)",         // Output de build
      "Read(*.key)",              // Archivos de clave
      "Read(*.pem)"              // Certificados
    ]
  }
}

Que archivos tipicamente excluir

Tipo Patron Razon
Secrets.env, .env.*Variables con API keys y passwords
Credencialessecrets/**, *.key, *.pemClaves privadas y certificados
Dependenciasnode_modules/**, vendor/**Demasiados archivos, no relevantes
Build outputbuild/**, dist/**Archivos generados
⚠ Limitacion importante

Las reglas deny previenen que Claude lea el archivo, pero no es una frontera de seguridad dura. Para secrets verdaderamente sensibles, usa un secrets manager (no dependas solo de deny rules).

respectGitignore

El setting respectGitignore (default true) hace que el file picker @ respete los patrones de .gitignore. Los archivos que matchean se excluyen de las sugerencias de autocompletado.

ⓘ Alternativa comunitaria

Existe el paquete claude-ignore que implementa un hook PreToolUse con sintaxis tipo .gitignore, pero no es oficial de Anthropic.

2.4 — keybindings.json

Ubicacion: ~/.claude/keybindings.json. Ejecuta /keybindings para crear o abrir el archivo. Los cambios se aplican automaticamente sin reiniciar.

Formato del archivo

{
  "$schema": "https://www.schemastore.org/claude-code-keybindings.json",
  "bindings": [
    {
      "context": "Chat",
      "bindings": {
        "ctrl+e": "chat:externalEditor",
        "ctrl+s": "chat:stash",
        "ctrl+u": null          // Desactivar atajo
      }
    }
  ]
}

Atajos mas utiles

Accion Default Que hace
chat:submitEnterEnviar mensaje
chat:newline(sin asignar)Nueva linea sin enviar
chat:cycleModeShift+TabCambiar modo de permisos
chat:modelPickerCmd+PSelector de modelo
chat:externalEditorCtrl+GAbrir prompt en editor externo
chat:fastModeMeta+OToggle fast mode
chat:stashCtrl+SGuardar prompt actual
chat:killAgentsCtrl+X Ctrl+KMatar agentes en background
app:toggleTodosCtrl+TToggle lista de tareas
app:toggleTranscriptCtrl+OToggle transcript verboso
history:searchCtrl+RBuscar en historial
task:backgroundCtrl+BEnviar tarea a background

Contextos disponibles

Cada atajo pertenece a un contexto que define cuando esta activo:

// Contextos principales:
"Global"          // Siempre activo
"Chat"            // En el input de chat
"Autocomplete"    // Cuando hay sugerencias
"Confirmation"    // En prompts de permisos
"HistorySearch"   // Buscando en historial
"Task"            // En vista de tareas
"DiffDialog"      // Revisando diffs
"ModelPicker"     // Seleccionando modelo

Chords (secuencias de teclas)

// Secuencia: Ctrl+K luego Ctrl+S (separado por espacio)
"ctrl+k ctrl+s": "chat:submit"

// Reclamar un prefijo: desactivar chords y reasignar
{
  "context": "Chat",
  "bindings": {
    "ctrl+x ctrl+k": null,     // Liberar chord
    "ctrl+x ctrl+e": null,     // Liberar chord
    "ctrl+x": "chat:newline"    // Reasignar prefijo
  }
}

Teclas reservadas (no se pueden reasignar)

Atajo Razon
Ctrl+CInterrupt hardcodeado en la terminal
Ctrl+DExit hardcodeado en la terminal
Ctrl+MIdentico a Enter en terminales
ⓘ Diagnostico

Ejecuta /doctor para ver advertencias de keybindings: errores de parseo, contextos invalidos, conflictos con teclas reservadas y duplicados.

2.5 — Estructura completa de archivos

A nivel de proyecto

proyecto/
  CLAUDE.md                  # Instrucciones (alternativa: .claude/CLAUDE.md)
  .mcp.json                  # Servidores MCP del proyecto
  .claude/
    CLAUDE.md              # Instrucciones del proyecto
    settings.json          # Settings compartidos (commitear a git)
    settings.local.json    # Settings locales (gitignored)
    rules/                 # Instrucciones modulares
      code-style.md
      testing.md
      security.md
    agents/                # Definiciones de subagentes custom
    commands/              # Slash commands personalizados

A nivel de usuario

~/.claude/
  CLAUDE.md                  # Instrucciones personales (todos los proyectos)
  settings.json              # Settings de usuario
  keybindings.json           # Atajos de teclado personalizados
  rules/                     # Reglas personales (todos los proyectos)
    preferences.md
    workflows.md
  agents/                    # Subagentes de usuario
  commands/                  # Commands de usuario
  projects/                  # Auto memoria por proyecto
    <hash-proyecto>/
      memory/
        MEMORY.md            # Indice de memorias
        topic-file.md        # Archivos por tema

A nivel enterprise (managed)

# macOS
/Library/Application Support/ClaudeCode/
  managed-settings.json      # Settings que nadie puede sobreescribir
  managed-settings.d/        # Fragmentos de diferentes equipos
    10-telemetry.json
    20-security.json
  CLAUDE.md                  # Instrucciones enterprise obligatorias
  managed-mcp.json           # MCP servers administrados

# Linux / WSL
/etc/claude-code/
  managed-settings.json
  CLAUDE.md

# Windows
C:\Program Files\ClaudeCode\
  managed-settings.json
  CLAUDE.md
✔ Buena practica — Organizacion clara

Usa esta regla simple para decidir donde poner cada cosa:

  • Afecta al equipo? → .claude/settings.json + CLAUDE.md (commiteados)
  • Solo a ti? → .claude/settings.local.json + ~/.claude/
  • A toda la empresa? → Managed settings (deploy por IT)
✘ Mala practica

Poner preferencias personales (modelo, effort, exclusiones) en .claude/settings.json que se commitea a git. Tus companeros terminan con tu configuracion sin quererlo.

Resumen rapido

CLAUDE.md

Instrucciones en lenguaje natural. Maximo ~200 lineas. Especifico y verificable. Soporta @imports y rules/ modulares.

settings.json

Configuracion tecnica: permisos, hooks, modelo, env. Separar shared (.claude/) de personal (~/.claude/ y .local).

.claudeignore

No existe nativamente. Usa reglas deny con Read(path) en settings.json.

keybindings.json

En ~/.claude/keybindings.json. Soporta chords, contextos y se aplica sin reiniciar. Usa /doctor para diagnostico.
