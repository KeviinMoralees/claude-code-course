Modulo 3 de 11

Manejo del Contexto

Ventana de contexto, que lo consume, compresion, comandos y mejores practicas

3.1 — La ventana de contexto

La ventana de contexto es la memoria de trabajo total de una sesion de Claude Code. Contiene absolutamente todo: system prompt, CLAUDE.md, memoria, historial de conversacion, resultados de herramientas, archivos leidos, output de comandos. Todo compite por el mismo presupuesto finito de tokens.

⚠ Principio fundamental

"La ventana de contexto es el recurso mas importante a gestionar. El rendimiento degrada conforme se llena." — Documentacion oficial de Anthropic

Limites de tokens

Configuracion Tamano Disponibilidad
Default 200,000 tokens Todos los planes
Extended (1M) 1,000,000 tokens Opus 4.6 y Sonnet 4.6 (segun plan)

Que se carga al inicio (~tokens)

Antes de que escribas tu primer mensaje, la ventana ya tiene contenido:

Componente ~Tokens Visible? Notas
System prompt ~4,200 Oculto Instrucciones base de comportamiento y herramientas
Auto memory (MEMORY.md) ~680 Oculto Primeras 200 lineas o 25KB
Environment info ~280 Oculto Directorio, plataforma, shell, OS, git status
MCP tools (solo nombres) ~120 Oculto Schemas completos se cargan bajo demanda
Skill descriptions ~450 Oculto Solo descripciones de 1 linea; contenido on-demand
~/.claude/CLAUDE.md ~320 Oculto Preferencias globales del usuario
CLAUDE.md del proyecto ~1,800 Oculto Convenciones, build commands, arquitectura
Total al arrancar ~8,000-10,000 — Antes de que escribas nada
ⓘ Visualizacion interactiva

Anthropic ofrece una visualizacion interactiva del contexto en code.claude.com/docs/en/context-window donde puedes ver exactamente cuantos tokens consume cada componente.

3.2 — Que consume contexto

De mayor a menor volumen tipico

  1. Lecturas de archivos — Un solo Read puede consumir miles de tokens (ej: auth.ts = ~2,400 tokens)
  2. Output de comandos — npm test = ~1,200 tokens, Grep = ~600 tokens
  3. Respuestas de Claude — Analisis, explicaciones, codigo
  4. Historial de conversacion — Cada mensaje se acumula
  5. CLAUDE.md — Se carga cada sesion
  6. Definiciones de MCP tools — Pueden ser significativas
  7. Output de hooks — additionalContext entra sin truncar
  8. Skills — Solo cuando se invocan (on-demand)
  9. Path-specific rules — Se cargan al leer archivos que matchean

Impacto de cada herramienta en el contexto

Herramienta Tokens tipicos Lo que tu ves Lo que Claude ve
Read 1,000 - 3,000+ "Read auth.ts" Las 2,400 tokens completas del archivo
Grep 400 - 1,000+ El comando que corrio Todas las lineas que matchearon
Glob 100 - 500 Listado de archivos Todos los paths encontrados
Edit 200 - 600 El diff visual Parametros del edit + resultado
Write 200 - 1,000+ "Wrote file.ts" Contenido completo del archivo
Bash Varia mucho El output truncado Puede ser enorme (test suites, logs)
⚠ Asimetria clave

Tu ves un resumen de 1 linea en la terminal, pero Claude ve el output completo de cada herramienta. Un simple npm test que tu ves como "tests passed" consume ~1,200 tokens internamente.

Especificos del tool Read

Parametro Descripcion Default
limit Numero de lineas a leer 2,000 lineas
offset Linea desde donde empezar 0 (inicio)
pages Rango de paginas para PDFs (ej: "1-5") Todas (max 20 por request)

Formatos soportados:

  • Codigo — Cualquier archivo de texto, con numeros de linea (cat -n)
  • Imagenes — PNG, JPG, etc. (contenido visual multimodal)
  • PDFs — Para mas de 10 paginas, requiere parametro pages. Max 20 paginas/request
  • Jupyter notebooks — .ipynb con todas las celdas y outputs
✔ Buena practica — Leer solo lo necesario
# BIEN: leer solo las lineas que necesitas
# "Lee las lineas 50 a 100 de auth.ts" (~50 lineas = ~200 tokens)

# BIEN: leer solo las paginas relevantes de un PDF
# Read con pages: "3-5" (~3 paginas)
✘ Mala practica — Leer archivos completos sin necesidad
# MAL: "lee todo el archivo package-lock.json"
# Puede ser 50,000+ tokens que llenan 25% del contexto de golpe

# MAL: "lee este PDF de 200 paginas"
# Falla porque el maximo es 20 paginas por request

✔ Buena practica — @-mentions para precision
# Incluir archivos especificos en tu prompt
Revisa @src/auth/middleware.ts y @src/auth/types.ts
y dime si hay vulnerabilidades de seguridad.

# Incluir una carpeta completa
Analiza la arquitectura de @src/services/

Los @-mentions cargan exactamente lo que necesitas, sin que Claude tenga que buscar.

✘ Mala practica — Prompts vagos que causan lecturas masivas
# MAL: esto hace que Claude lea docenas de archivos buscando
"Mejora el codigo del proyecto"

# MAL: pipe de un log gigante
cat /var/log/syslog | claude -p "que paso?"
# Puede ser 100,000+ tokens de golpe

Prompts vagos generan muchas lecturas exploratorias que llenan el contexto rapidamente.

3.4 — Compresion automatica (Auto-compaction)

Se activa automaticamente cuando te acercas al limite de la ventana de contexto. Funciona en 2 etapas:

Etapa 1 — Limpieza de tool results

Elimina outputs crudos de herramientas de mensajes viejos. Es una operacion ligera porque los agentes tipicamente no necesitan re-leer resultados antiguos.

Etapa 2 — Resumen de conversacion

Si la limpieza no libera suficiente espacio, comprime toda la conversacion en un resumen estructurado.

Que sobrevive vs que se pierde

Sobrevive SIEMPRE Se preserva en resumen Puede perderse
CLAUDE.md (re-leido del disco)
Auto memory (persiste fuera del contexto)
System prompt 		Decisiones arquitectonicas
Bugs no resueltos
Detalles de implementacion
Tus requests clave
Code snippets importantes
5 archivos mas recientes 		Instrucciones del inicio de la conversacion
Outputs de herramientas redundantes
Output verboso de comandos
Razonamiento intermedio
Listado de skill descriptions
⚠ Dato critico

Si diste una instruccion importante en la conversacion (no en CLAUDE.md), puede perderse durante la compactacion. Todo lo que debe sobrevivir siempre va en CLAUDE.md, no en mensajes de chat.

✔ Buena practica — Proteger instrucciones criticas
# EN CLAUDE.md (sobrevive compactacion)
# Convenciones
- Usar Bun, no npm
- Correr bun test antes de cada commit
- No usar any en TypeScript

# NO en un mensaje de chat
# "Oye recuerda siempre usar Bun y no npm"
# Esto se pierde cuando el contexto se compacta
✘ Mala practica — Instrucciones efimeras para reglas permanentes
# MAL: instruccion importante solo en chat
Tu: "De ahora en adelante, todos los componentes deben usar
     el patron Container/Presenter"

# 30 minutos despues, tras compactacion...
Claude: *ya no recuerda la instruccion y vuelve a su estilo default*

Si Claude "olvida" algo que le dijiste, es porque estaba en conversacion y se compacto. Ponlo en CLAUDE.md.

3.5 — Comandos de contexto

Comando Que hace Cuando usarlo
/compact [instrucciones] Comprime historial en resumen estructurado Sesion larga, contexto lleno, antes de tarea nueva
/clear Reset total del contexto Cambiar a tarea completamente diferente
/context Muestra que consume espacio en el contexto Diagnosticar contexto lleno o lento
/mcp MCP servers y su costo en contexto Ver si un MCP server consume demasiado
/cost Estadisticas de tokens de la sesion Monitorear gasto (API users)
/rename Nombrar sesion actual Antes de /clear para poder reanudar despues

/compact con instrucciones de foco

Puedes dirigir que preservar durante la compactacion:

# Compactar enfocandose en cambios del API
/compact focus on the API changes

# Compactar preservando errores y testing
/compact enfocate en los cambios de testing y los errores encontrados

# Compactar preservando codigo especifico
/compact Focus on code samples and API usage

Instrucciones de compactacion permanentes en CLAUDE.md

# En tu CLAUDE.md del proyecto

# Compact instructions
When you are using compact, please focus on:
- Test output and failures
- Code changes and their reasoning
- Architectural decisions made

Esto aplica en toda compactacion (manual y automatica) sin tener que repetirlo.

Re-inyectar contexto post-compactacion con hooks

// En settings.json
{
  "hooks": {
    "SessionStart": [{
      "matcher": "compact",
      "hooks": [{
        "type": "command",
        "command": "echo 'Reminder: usar Bun, no npm. Correr bun test antes de commit.'"
      }]
    }]
  }
}

Este hook se ejecuta automaticamente despues de cada compactacion, re-inyectando informacion critica.

✔ Buena practica — Flujo de trabajo con /rename y /clear
# 1. Terminaste una tarea, nombra la sesion
/rename refactor-auth-module

# 2. Limpia el contexto para la siguiente tarea
/clear

# 3. Mas tarde, puedes reanudar si necesitas
claude -r refactor-auth-module

Nunca pierdes historial y empiezas cada tarea con contexto limpio.

✘ Mala practica — Sesion infinita
# MAL: usar la misma sesion para TODO durante horas
Tu: "arregla el bug de login"
Tu: "ahora agrega la feature de notificaciones"
Tu: "ahora refactoriza la base de datos"
Tu: "ahora haz el deploy"
# El contexto esta lleno de basura de tareas anteriores
# Claude responde mas lento y menos preciso

Cada tarea nueva debe empezar con /clear o al menos /compact. El contexto viejo es ruido.

3.6 — Contexto extendido (1M tokens)

Disponibilidad por plan

Plan Opus 1M Sonnet 1M
Max, Team, Enterprise Incluido (auto-upgrade) Requiere extra usage
Pro Requiere extra usage Requiere extra usage
API / pay-as-you-go Acceso completo Acceso completo

En Max/Team/Enterprise, Opus se upgradea automaticamente a 1M sin configuracion extra.

Como activar y desactivar

# Activar via model picker
/model opus[1m]
/model sonnet[1m]

# Fijar via variable de entorno
export ANTHROPIC_DEFAULT_OPUS_MODEL='claude-opus-4-6[1m]'

# Desactivar 1M completamente
export CLAUDE_CODE_DISABLE_1M_CONTEXT=1
ⓘ Precio

Sin premium por tokens mas alla de 200K. El pricing es el estandar del modelo. Para planes con suscripcion, el uso esta cubierto; para extra usage, se cobra normalmente.

⚠ Rendimiento

Tener 1M tokens no significa que debas llenarlos. El rendimiento del LLM degrada conforme el contexto crece (limitacion arquitectonica de transformers, "context rot"). La auto-compactacion sigue operando dentro del 1M, solo con un techo mas alto. La regla sigue siendo: mantener el contexto relevante y compacto.

3.7 — Las 10 mejores practicas

1. /clear entre tareas

Contexto viejo de tareas anteriores es ruido. Cada tarea nueva merece contexto limpio.

2. Prompts especificos

"Arregla el bug en auth.ts linea 42" causa menos lecturas que "mejora el codebase".

3. CLAUDE.md < 200 lineas

Mueve workflows detallados a skills (on-demand) o .claude/rules/ (path-scoped).

4. Subagentes para investigacion

Un subagente lee 6,100 tokens pero devuelve solo ~420 al contexto principal.

5. Plan mode primero

Explora y planifica antes de implementar. Evita re-trabajo costoso en contexto.

6. Hooks para procesar datos

Un hook puede grep 10,000 lineas y devolver solo las relevantes. De miles a cientos de tokens.

7. CLI tools sobre MCP

gh, aws, gcloud no agregan overhead de definicion de tools al contexto.

8. Desactivar MCP no usados

Cada MCP server agrega tokens al contexto. Usa /mcp para ver y desactivar.

9. Reglas permanentes en CLAUDE.md

Lo que debe sobrevivir compactacion va en CLAUDE.md, nunca en mensajes de chat.

10. Guardar en memoria

En sesiones largas, guarda progreso en auto memory antes de que el contexto se comprima.

✔ Flujo optimo de una sesion larga
# 1. Empezar con contexto limpio
/clear

# 2. Planificar antes de implementar
/plan # o pedir un plan en modo plan

# 3. Implementar (Claude lee y edita archivos)

# 4. Si el contexto se llena, compactar con foco
/compact enfocate en los cambios del modulo auth

# 5. Verificar que esta consumiendo contexto
/context

# 6. Antes de cambiar de tarea
/rename tarea-completada
/clear
✘ Anti-patron: el "mega prompt"
# MAL: cargar CLAUDE.md de 500 lineas + 4 MCP servers
# + leer 15 archivos de golpe + pedir "mejora todo"

Tu: "Lee @src/ @lib/ @tests/ @docs/ @config/
     y mejora la calidad general del codigo,
     agrega tests faltantes, actualiza la documentacion
     y optimiza el rendimiento."

# Resultado: contexto lleno en el primer turno,
# Claude no puede ni empezar a trabajar efectivamente

Divide tareas grandes en pasos discretos. Cada paso con su propio contexto enfocado.
