11.2 — Messages API

El endpoint central de la API de Claude. Soporta conversaciones multi-turno, contenido mixto (texto, imágenes, documentos), tool use y streaming.

Request mínimo

Parámetros principales

Parámetro Tipo Req Descripción
model string ✓ ID del modelo a usar
messages array ✓ Array de objetos {role, content} — roles: user / assistant
max_tokens number ✓ Máximo de tokens a generar. Límite varía por modelo.
system string/array - System prompt. Usar el parámetro top-level, no como mensaje con role "system".
tools array - Definiciones de herramientas disponibles para Claude
stream boolean - Habilitar streaming SSE (default: false)
temperature number - Aleatoriedad 0.0–1.0 (default: 1.0)
thinking object - Extended thinking: {"type": "enabled", "budget_tokens": N}
stop_sequences array - Secuencias que detienen la generación

11.3 — Streaming

Con stream: true, la API envía eventos Server-Sent Events (SSE) en tiempo real. Ideal para UIs que muestran la respuesta mientras se genera.

Eventos de streaming

Evento Cuándo ocurre Contiene
message_start Inicio de respuesta ID, modelo, usage inicial
content_block_start Nuevo bloque de contenido Tipo del bloque (text, thinking, tool_use)
content_block_delta Fragmento incremental text_delta, thinking_delta, input_json_delta
content_block_stop Bloque completo —
message_delta Metadata final stop_reason, usage final
message_stop Stream terminado —

11.4 — Tool Use (Function Calling)

Claude puede llamar a herramientas cuando lo necesite para responder. Existen dos tipos: client tools (tú las ejecutas) y server tools (Anthropic las ejecuta).

Client tools vs Server tools

Tipo Quién ejecuta Flujo Ejemplos
Client tools Tu aplicación Claude responde con tool_use → tú ejecutas → envías tool_result APIs propias, base de datos, lógica interna
Server tools Anthropic Claude llama y recibe resultado directamente web_search, code_execution, web_fetch

11.6 — Extended Thinking

El extended thinking permite que Claude razone paso a paso antes de responder, con visibilidad del proceso de razonamiento interno. Ideal para problemas complejos.

Cuándo usar extended thinking

  • Matemáticas complejas y lógica formal
  • Decisiones arquitectónicas con múltiples trade-offs
  • Debugging de problemas difíciles y no obvios
  • Análisis multi-paso con inferencias encadenadas

Habilitar extended thinking

Parámetros del thinking

Parámetro Descripción
budget_tokens Tokens máximos para razonamiento interno. Mínimo: 1024. Debe ser menor que max_tokens. Encima de 32k hay rendimientos decrecientes.
display: "summarized" Default. Muestra un resumen del pensamiento en la respuesta.
display: "omitted" No incluye el thinking en la respuesta — menor latencia para apps que no lo muestran. Se cobra igual.

Se cobran los tokens de thinking completo, no los del resumen visible. El campo usage.output_tokens puede no coincidir con el texto visible en la respuesta. Para Opus 4.6 y Sonnet 4.6, el razonamiento adaptativo ya está incluido — usar budget_tokens solo en modelos más antiguos.

11.7 — Construir Aplicaciones

Chatbot con historial de conversación

Pipeline de procesamiento con herramientas

Prompt caching para reducir costos

Batch API para procesamiento masivo

  • Usar el SDK en lugar de llamadas HTTP directas — maneja retry automático con backoff exponencial
  • Siempre especificar max_tokens según el output esperado — evita cargos inesperados
  • Usar cache_control para system prompts y documentos que se reusan entre requests
  • Para outputs estructurados, usar tool use con strict: true en lugar de pedir JSON en el prompt
  • Batch API para tareas no urgentes de más de 100 requests — hasta 50% de ahorro
  • Manejar rate limits (429) con retry — el SDK lo hace automáticamente
  • Hardcodear la API key en el código — usar variables de entorno o secrets managers
  • No manejar stop_reason: "max_tokens" — la respuesta puede estar truncada
  • Pedir JSON en el prompt en lugar de usar tool use — más frágil e impredecible
  • Enviar el historial completo sin límite — el contexto crece indefinidamente
  • Usar Opus para tareas simples — Haiku o Sonnet son 5x–15x más baratos
  • No usar prompt caching en aplicaciones con system prompts repetitivos
