6.1 — Commands vs Skills: diferencia clave

Son dos conceptos distintos. Confundirlos lleva a errores de configuracion y expectativas incorrectas.

Commands built-in

Son comandos hardcoded en el binario de Claude Code. No los creas, no los modificas, no viven en archivos. Ejecutan logica fija del sistema cuando los invocas con /.

Comando Funcion
/help Muestra ayuda general de Claude Code
/compact [focus] Compacta el contexto con instruccion de foco opcional
/clear Limpia la conversacion completa
/context Muestra el uso actual del contexto por categoria
/mcp Lista y administra servidores MCP activos
/permissions Ver y editar reglas de permisos de la sesion
/init Genera un CLAUDE.md para el proyecto actual
/cost Muestra el costo de tokens de la sesion
/doctor Diagnostico de instalacion y configuracion
/fast Alterna el modo rapido (Fast mode)
/memory Ver y editar archivos de memoria del sistema
/skills Listar skills disponibles en la sesion actual
Los Commands built-in son de Claude Code, no tuyos. Las Skills son tuyas: las creas, las configuras y Claude las usa automaticamente o cuando las invocas.

Skills — el sistema moderno

Las Skills son archivos en disco que empaquetan instrucciones, recursos y codigo ejecutable. Claude las descubre y carga bajo demanda segun relevancia. Son el reemplazo y evolucion de los "custom commands" legacy (.claude/commands/).

El sistema antiguo usaba .claude/commands/deploy.md para crear /deploy. Sigue funcionando, pero Skills es el sistema unificado que lo reemplaza con mucho mas poder. Un archivo en .claude/commands/deploy.md y una skill en .claude/skills/deploy/SKILL.md producen el mismo /deploy — pero la skill admite frontmatter, archivos de soporte, scripts y control de invocacion.

6.2 — Arquitectura de una Skill

Estructura de directorios

Anatomia de SKILL.md

Los tres niveles de carga (Progressive Disclosure)

Nivel Cuando carga Costo tokens Contenido
1 — Metadata Siempre, al inicio ~100 tokens por skill name y description del YAML
2 — Instrucciones Cuando la skill es relevante <5k tokens Cuerpo completo de SKILL.md
3 — Recursos Solo si se necesitan Sin limite practico Archivos extra, scripts (solo output al ejecutar)
Puedes tener decenas de skills instaladas sin penalizar el contexto. Solo la metadata (nombre + descripcion) de todas las skills vive en el sistema prompt. El cuerpo de SKILL.md solo se carga cuando Claude decide que esa skill es relevante para tu tarea. Los scripts nunca entran al contexto — solo su output.

6.3 — Donde viven las Skills

Alcance Ruta Se aplica a
Enterprise Managed settings (admin) Todos los usuarios de la organizacion
Personal ~/.claude/skills/<skill>/SKILL.md Todos tus proyectos
Proyecto .claude/skills/<skill>/SKILL.md Solo este proyecto
Plugin <plugin>/skills/<skill>/SKILL.md Donde el plugin este activo

Cuando skills comparten el mismo nombre en distintos niveles, gana la mayor precedencia: enterprise > personal > proyecto > plugin.

Claude Code descubre skills automaticamente en subdirectorios. Si tienes packages/api/.claude/skills/, las skills ahi se activan cuando trabajas con archivos de ese subdirectorio.

6.4 — Skills bundled (incluidas en Claude Code)

Estas skills vienen preinstaladas y disponibles en todas las sesiones. A diferencia de los commands built-in, estan implementadas como skills de prompt: le dan a Claude un playbook detallado y dejan que orqueste el trabajo con sus herramientas.

Skill Proposito
/batch <instruccion> Cambios masivos en paralelo. Descompone el trabajo en 5-30 unidades independientes, crea un plan, luego lanza un agente por unidad en worktrees aislados. Cada agente implementa, corre tests y abre un PR. Requiere repositorio git.
/claude-api Carga referencia de la API de Claude para el lenguaje del proyecto (Python, TS, Go, Java, Ruby, C#, PHP, cURL). Se activa automaticamente cuando detecta imports de anthropic o @anthropic-ai/sdk.
/debug [descripcion] Activa debug logging para la sesion actual y analiza el log para diagnosticar problemas. El debug esta desactivado por default a menos que hayas iniciado con claude --debug.
/loop [intervalo] <prompt> Ejecuta un prompt repetidamente mientras la sesion este abierta. Util para polling de deploys, monitoreo de PRs, tareas periodicas. Ej: /loop 5m check if the deploy finished.
/simplify [foco] Lanza 3 agentes de revision en paralelo sobre los archivos recien modificados, agrega sus hallazgos y aplica correcciones de reutilizacion, calidad y eficiencia. Ej: /simplify focus on memory efficiency.

6.7 — Contexto dinamico con !`command`

La sintaxis !`comando` ejecuta comandos shell antes de que Claude vea el contenido de la skill. El output reemplaza el placeholder. Esto es preprocesamiento, no algo que Claude ejecuta.

Para activar razonamiento extendido (ultrathink) dentro de una skill, incluye la palabra ultrathink en cualquier parte del contenido de SKILL.md.

6.8 — Skills en subagentes (context: fork)

Con context: fork, el contenido de SKILL.md se convierte en el prompt que maneja un subagente aislado. El subagente no tiene acceso al historial de la conversacion.

Enfoque System prompt Task Tambien carga
Skill con context: fork Del tipo de agente (Explore, Plan...) Contenido de SKILL.md CLAUDE.md
Subagente con campo skills Markdown del subagente Mensaje de delegacion de Claude Skills precargadas + CLAUDE.md
context: fork solo tiene sentido para skills con instrucciones de tarea explicitas. Si tu skill solo contiene guias o conocimiento de fondo sin una tarea concreta, el subagente retornara sin hacer nada util.

6.9 — Buenas practicas (documentacion oficial)

Principio 1: Conciso es clave

El contexto es un bien compartido. Tu skill compite por tokens con el historial de conversacion, otras skills y la solicitud actual. Solo agrega lo que Claude no sabe ya.

Claude ya sabe que es un PDF y como funcionan las librerias. No lo expliques.

Principio 2: Nivel de libertad apropiado

Ajusta la especificidad de las instrucciones segun la fragilidad de la tarea.

Principio 3: Nombres y descripciones efectivas

Claude no puede saber cuando usar esta skill vs otra que tambien "ayuda con documentos".

Reglas para nombres: usa gerundio (processing-pdfs, analyzing-spreadsheets, managing-databases). Evita nombres vagos como helper, utils, tools.

Reglas para descripciones: escribe en tercera persona ("Processes Excel files" — no "I can help you"). Incluye tanto que hace la skill como cuando usarla. Las descripciones se truncan a 250 caracteres si hay muchas skills instaladas, asi que pon lo mas importante primero.

Principio 4: Divulgacion progresiva

Meter 800 lineas de documentacion, ejemplos, referencia de API y casos edge en un solo SKILL.md. Esto carga todo en contexto aunque solo se necesite el 10%.

Mantener SKILL.md bajo 500 lineas. Las referencias deben ser maximo 1 nivel de profundidad desde SKILL.md.

Principio 8: Desarrollo iterativo con evaluaciones

Crea evaluaciones antes de escribir documentacion extensa. El proceso es:

  1. Ejecuta la tarea sin skill — identifica que falla
  2. Crea 3 evaluaciones que prueben esos gaps
  3. Mide el baseline de Claude sin skill
  4. Escribe el minimo de instrucciones para cubrir los gaps
  5. Itera: Claude A crea/refina la skill, Claude B la prueba en tareas reales

6.10 — Anti-patrones a evitar

Dale a Claude una respuesta, no una pregunta. Un default claro con una escotilla de escape para casos especiales.

Siempre usa forward slashes, incluso en Windows.
