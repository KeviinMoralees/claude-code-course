Modulo 8 de 11

MCP — Model Context Protocol

Conecta Claude Code a herramientas externas: bases de datos, APIs, issue trackers, monitoreo y mas

8.1 — Que es MCP

MCP es un protocolo abierto para integrar herramientas externas con Claude. Un servidor MCP expone tools, resources y prompts que Claude puede usar dentro de una sesion. Una vez conectado, Claude tiene acceso directo a tus sistemas.

Lo que puedes hacer con MCP

Issue trackers

"Implementa el feature del JIRA ENG-4521 y crea un PR en GitHub"

Monitoreo

"Revisa Sentry y Statsig para ver el uso del feature ENG-4521"

Bases de datos

"Encuentra clientes que no han comprado en 90 dias en PostgreSQL"

Diseno y comunicacion

"Actualiza el template de email con los nuevos disenos de Figma posteados en Slack"

Automatizacion

"Crea borradores en Gmail invitando a estos 10 usuarios a una sesion de feedback"

Eventos externos (Channels)

Un servidor MCP puede hacer push de mensajes a tu sesion: reaccionar a Telegram, Discord, webhooks mientras no estas

Seguridad

Usa servidores MCP de terceros bajo tu propio riesgo. Anthropic no ha verificado la seguridad de todos los servidores disponibles. Ten especial cuidado con servidores que consumen contenido externo — pueden exponerte a prompt injection. Solo instala servidores de fuentes en las que confias.

8.2 — Instalar servidores MCP

Opcion 1: HTTP (recomendado para servidores remotos)

# Sintaxis basica
claude mcp add --transport http <nombre> <url>

# Ejemplo: Notion
claude mcp add --transport http notion https://mcp.notion.com/mcp

# Con header de autenticacion Bearer
claude mcp add --transport http secure-api https://api.example.com/mcp \
  --header "Authorization: Bearer your-token"

Opcion 2: SSE (deprecado)

# El transporte SSE esta deprecado — usa HTTP cuando sea posible
claude mcp add --transport sse asana https://mcp.asana.com/sse

Opcion 3: Stdio (procesos locales)

# El -- separa las opciones de Claude del comando del servidor MCP
claude mcp add --transport stdio \
  --env AIRTABLE_API_KEY=YOUR_KEY \
  airtable -- npx -y airtable-mcp-server

# PostgreSQL read-only
claude mcp add --transport stdio db -- npx -y @bytebase/dbhub \
  --dsn "postgresql://readonly:pass@prod.db.com:5432/analytics"

# Sentry via HTTP
claude mcp add --transport http sentry https://mcp.sentry.dev/mcp

# GitHub
claude mcp add --transport http github https://api.githubcopilot.com/mcp/
Orden de opciones — importante

Todas las opciones (--transport, --env, --scope, --header) deben ir antes del nombre del servidor. El -- separa el nombre del servidor del comando que se pasa al servidor MCP. Esto evita conflictos entre los flags de Claude y los del servidor.

Windows (sin WSL)

En Windows nativo, los servidores stdio con npx requieren el wrapper cmd /c: 

claude mcp add --transport stdio my-server -- cmd /c npx -y @some/package

Sin esto obtienes errores "Connection closed".

8.3 — Comandos de gestion

# Ver todos los servidores configurados
claude mcp list

# Detalles de un servidor especifico
claude mcp get github

# Eliminar un servidor
claude mcp remove github

# Agregar desde config JSON directamente
claude mcp add-json weather-api '{"type":"http","url":"https://api.weather.com/mcp"}'

# Importar servidores desde Claude Desktop (macOS / WSL)
claude mcp add-from-claude-desktop

# Dentro de Claude Code: ver estado, autenticar con OAuth
/mcp

# Verificar timeout de conexion (default: sin limite explicito)
MCP_TIMEOUT=10000 claude  # timeout de 10 segundos

8.4 — Scopes de instalacion

Scope Donde se guarda Para que
local (default) ~/.claude.json bajo la ruta del proyecto Personal, credentials sensibles, experimental para este proyecto
project .mcp.json en el root del proyecto Compartido con el equipo via git, herramientas del proyecto
user ~/.claude.json global Herramientas personales que usas en todos los proyectos 
claude mcp add --transport http stripe --scope local   https://mcp.stripe.com
claude mcp add --transport http paypal --scope project https://mcp.paypal.com/mcp
claude mcp add --transport http hubspot --scope user   https://mcp.hubspot.com/anthropic

Precedencia: local > project > user. Si un servidor esta configurado en local y ademas como conector de Claude.ai, gana la configuracion local.

Formato de .mcp.json

{
  "mcpServers": {
    "shared-server": {
      "command": "/path/to/server",
      "args": [],
      "env": {}
    }
  }
}

Variables de entorno en .mcp.json

{
  "mcpServers": {
    "api-server": {
      "type": "http",
      "url": "${API_BASE_URL:-https://api.example.com}/mcp",
      "headers": {
        "Authorization": "Bearer ${API_KEY}"
      }
    }
  }
}

Sintaxis: ${VAR} expande la variable. ${VAR:-default} usa el default si VAR no esta definida. Se puede usar en command, args, env, url y headers.

Nota sobre terminologia

El scope MCP "local" se guarda en ~/.claude.json (home directory), no en el directorio del proyecto. Esto es diferente a los settings generales de Claude Code donde "local" se refiere a .claude/settings.local.json dentro del proyecto.

8.5 — Autenticacion

OAuth 2.0 automatico

# 1. Agregar el servidor
claude mcp add --transport http sentry https://mcp.sentry.dev/mcp

# 2. Autenticar
/mcp  → seleccionar servidor → "Authenticate" → login en navegador

Los tokens se guardan de forma segura y se refrescan automaticamente. Para revocar: /mcp → "Clear authentication".

Puerto OAuth fijo (para redirect URIs pre-registradas)

claude mcp add --transport http \
  --callback-port 8080 \
  my-server https://mcp.example.com/mcp

Credenciales OAuth pre-configuradas

# Cuando el servidor no soporta Dynamic Client Registration
claude mcp add --transport http \
  --client-id your-client-id --client-secret --callback-port 8080 \
  my-server https://mcp.example.com/mcp

# Via variable de entorno (para CI/CD)
MCP_CLIENT_SECRET=your-secret claude mcp add --transport http \
  --client-id your-client-id --client-secret --callback-port 8080 \
  my-server https://mcp.example.com/mcp

Headers dinamicos (Kerberos, tokens efimeros, SSO interno)

// .mcp.json
{
  "mcpServers": {
    "internal-api": {
      "type": "http",
      "url": "https://mcp.internal.example.com",
      "headersHelper": "/opt/bin/get-mcp-auth-headers.sh"
    }
  }
}

El script corre en cada conexion, escribe JSON de headers a stdout. Variables disponibles para el script: CLAUDE_CODE_MCP_SERVER_NAME y CLAUDE_CODE_MCP_SERVER_URL. Timeout: 10 segundos.

Un script para multiples servidores

Usando las variables CLAUDE_CODE_MCP_SERVER_NAME y CLAUDE_CODE_MCP_SERVER_URL, puedes escribir un solo headersHelper que sirva para todos tus servidores internos, generando el token correcto segun el servidor que lo solicite.

8.6 — Resources y @-menciones

Los servidores MCP pueden exponer recursos que se referencian en prompts con @, igual que referencias a archivos. 

# Analizar un issue de GitHub
Can you analyze @github:issue://123 and suggest a fix?

# Revisar documentacion
Please review @docs:file://api/authentication

# Comparar recursos de dos servidores
Compare @postgres:schema://users with @docs:file://database/user-model

Escribe @ en el prompt para ver autocompletado con todos los recursos disponibles de los servidores conectados. Los recursos se incluyen automaticamente como adjuntos al referenciarlos.

8.7 — MCP Prompts como comandos

Los servidores MCP pueden exponer prompts que aparecen como comandos / en Claude Code. Formato: /mcp__<servidor>__<prompt> 

# Sin argumentos
/mcp__github__list_prs

# Con argumentos (separados por espacio)
/mcp__github__pr_review 456
/mcp__jira__create_issue "Bug in login flow" high

Los espacios en nombres de servidores o prompts se convierten automaticamente en guiones bajos.

8.8 — Tool Search — carga diferida

Por defecto los tools de MCP se cargan bajo demanda, no todos al inicio. Solo los nombres se cargan en el system prompt. Claude busca los tools relevantes cuando los necesita. Esto permite tener decenas de servidores sin penalizar el contexto.

Valor ENABLE_TOOL_SEARCH Comportamiento
(no definido) Todos los tools diferidos y cargados bajo demanda. Carga upfront si ANTHROPIC_BASE_URL apunta a host no oficial
true Todos los tools diferidos, incluso con ANTHROPIC_BASE_URL no oficial
auto Modo umbral: carga upfront si caben en el 10% del contexto, diferidos en caso contrario
auto:N Modo umbral con porcentaje personalizado (ej: auto:5 para 5%)
false Todos los tools cargados upfront, sin deferral 
# Umbral personalizado del 5%
ENABLE_TOOL_SEARCH=auto:5 claude

# Desactivar Tool Search (cargar todo upfront)
ENABLE_TOOL_SEARCH=false claude

# Deshabilitar ToolSearch como herramienta en settings.json
{
  "permissions": { "deny": ["ToolSearch"] }
}
Modelos soportados

Tool Search requiere modelos que soporten bloques tool_reference: Sonnet 4+ y Opus 4+. Haiku no lo soporta.

Para autores de servidores MCP

El campo server instructions del servidor se vuelve mas valioso con Tool Search activo. Ayuda a Claude a saber cuando buscar tus tools. Explica que categoria de tareas maneja tu servidor y cuando usarlo. Claude Code trunca las tool descriptions e instrucciones del servidor a 2KB cada una.

8.9 — Limites de output y actualizaciones dinamicas

Limites de output

# Warning cuando un tool produce mas de 10,000 tokens
# Limite default: 25,000 tokens
# Para aumentar el limite:
MAX_MCP_OUTPUT_TOKENS=50000 claude

Actualizaciones dinamicas de tools

Claude Code soporta notificaciones list_changed de MCP. Cuando un servidor actualiza sus tools disponibles, Claude Code los refresca automaticamente sin necesidad de desconectar y reconectar el servidor.

8.10 — Claude Code como servidor MCP

Claude Code puede exponerse como servidor MCP para que otras aplicaciones lo usen. 

# Iniciar Claude como servidor MCP stdio
claude mcp serve

Configuracion para Claude Desktop

// claude_desktop_config.json
{
  "mcpServers": {
    "claude-code": {
      "type": "stdio",
      "command": "/full/path/to/claude",  // usa "which claude" para encontrar la ruta
      "args": ["mcp", "serve"],
      "env": {}
    }
  }
}
Ruta del ejecutable

El campo command debe apuntar al ejecutable de Claude Code. Si claude no esta en el PATH del sistema, usa la ruta completa obtenida con which claude. Sin esto obtienes errores spawn claude ENOENT.

8.11 — MCP gestionado (Enterprise)

Opcion 1: Control exclusivo con managed-mcp.json

El archivo toma control total de todos los servidores MCP. Los usuarios no pueden agregar ni modificar servidores.

Sistema Ruta del archivo
macOS/Library/Application Support/ClaudeCode/managed-mcp.json
Linux / WSL/etc/claude-code/managed-mcp.json
WindowsC:\Program Files\ClaudeCode\managed-mcp.json
 
// managed-mcp.json
{
  "mcpServers": {
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/"
    },
    "company-internal": {
      "type": "stdio",
      "command": "/usr/local/bin/company-mcp-server",
      "args": ["--config", "/etc/company/mcp-config.json"],
      "env": { "COMPANY_API_URL": "https://internal.company.com" }
    }
  }
}

Opcion 2: Allowlists y Denylists (control con flexibilidad)

Permite a los usuarios agregar servidores propios dentro de politicas definidas por el administrador.

// managed settings
{
  "allowedMcpServers": [
    { "serverName": "github" },                                        // por nombre
    { "serverCommand": ["npx", "-y", "@modelcontextprotocol/server-filesystem"] }, // por comando exacto
    { "serverUrl": "https://mcp.company.com/*" },                       // por URL con wildcard
    { "serverUrl": "https://*.internal.corp/*" }                         // wildcard en subdominio
  ],
  "deniedMcpServers": [
    { "serverName": "dangerous-server" },
    { "serverUrl": "https://*.untrusted.com/*" }
  ]
}
Campo allowedMcpServers Comportamiento
undefined (default) Sin restricciones — usuarios pueden configurar cualquier servidor
[] (array vacio) Lockdown total — nadie puede agregar servidores
Lista de entradas Solo se permiten servidores que coincidan por nombre, comando o URL
Reglas criticas de las politicas

Denylist tiene precedencia absoluta — si un servidor coincide con la denylist, queda bloqueado aunque este en la allowlist.

Los comandos stdio usan match exacto: mismo comando y todos los argumentos en orden. ["npx", "-y", "server"] no coincide con ["npx", "server"].

Las URLs soportan wildcards * para cualquier secuencia de caracteres.

8.12 — Buenas y malas practicas

Bien — scope correcto por caso de uso

Bases de datos con credenciales en --scope local, nunca en el proyecto. Herramientas del equipo (GitHub, Sentry) en --scope project via .mcp.json versionado en git. Herramientas personales de dev en --scope user.

Mal — credenciales en .mcp.json de proyecto
// .mcp.json — MAL: esto va a git con el token expuesto
{
  "mcpServers": {
    "api": {
      "env": { "API_KEY": "sk-real-api-key-aqui" }
    }
  }
}

Usa ${API_KEY} y define el valor real en variables de entorno del sistema, nunca en el archivo.

Bien — variables de entorno en .mcp.json
// .mcp.json — BIEN: el token viene del entorno
{
  "mcpServers": {
    "api": {
      "type": "http",
      "url": "${API_BASE_URL:-https://api.example.com}/mcp",
      "headers": { "Authorization": "Bearer ${API_KEY}" }
    }
  }
}
Bien — servidor con acceso de solo lectura

Para bases de datos o APIs de produccion, conecta con credenciales de solo lectura. Si el servidor MCP solo necesita leer datos, no le des permisos de escritura. Reduce el radio de impacto de un error o prompt injection.

Mal — instalar servidores MCP sin verificar

Un servidor MCP con acceso a tu filesystem y credenciales puede exfiltrar datos si esta comprometido o maliciosamente disenado. Verifica la fuente, revisa el codigo si es posible, y usa servidores solo de fuentes de confianza.
