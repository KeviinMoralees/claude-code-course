Modulo 4 de 11

Sistema de Memoria

Auto memory, tipos de memoria, MEMORY.md, que guardar y que no, mejores practicas

4.1 — Que es Auto Memory

Auto memory es un sistema donde Claude Code se escribe notas a si mismo mientras trabaja contigo, acumulando conocimiento entre sesiones sin esfuerzo manual. Requiere Claude Code v2.1.59+.

CLAUDE.md vs Auto Memory

Aspecto CLAUDE.md Auto Memory
Quien lo escribe Tu Claude
Que contiene Instrucciones y reglas Aprendizajes y patrones
Scope Proyecto, usuario u organizacion Por directorio de trabajo (repo git)
Carga al inicio Completo Primeras 200 lineas / 25KB de MEMORY.md
Sobrevive /compact Si (re-leido del disco) Si (persiste fuera del contexto)

Donde se almacena

~/.claude/projects/<proyecto>/memory/
  MEMORY.md              # Indice (NO es una memoria en si)
  user_profile.md        # Memoria individual
  feedback_testing.md    # Memoria individual
  project_migration.md   # Memoria individual
  reference_grafana.md   # etc.

El <proyecto> se deriva del root del repo git, asi que todos los worktrees y subdirectorios del mismo repo comparten la misma memoria. Es local a la maquina — no se comparte entre maquinas ni cloud.

Cuando Claude decide guardar una memoria

Claude no guarda algo en cada sesion. Evalua 4 criterios:

Recurrencia

Va a surgir de nuevo en futuras conversaciones?

Inferibilidad

Se puede deducir leyendo el codigo, o es no-obvio?

Estabilidad

Es probable que siga siendo verdad, o es efimero?

Especificidad

Es particular de este proyecto?

Como se carga la memoria

  1. Al inicio de sesion: Primeras 200 lineas / 25KB de MEMORY.md se inyectan al contexto
  2. Bajo demanda: Los topic files se leen solo cuando Claude los necesita
  3. Filtrado por relevancia: Un sub-agente examina tu query contra nombres y descripciones para decidir que cargar (max 5 archivos)
ⓘ Dato clave

Solo las primeras 200 lineas de MEMORY.md se cargan al inicio. Todo lo que exceda ese limite es invisible al arrancar la sesion. Por eso es critico mantener MEMORY.md conciso y mover el detalle a topic files.

4.2 — Los 4 tipos de memoria

1. User (usuario)

Informacion sobre el rol, metas, responsabilidades y conocimiento del usuario. Permite a Claude adaptar su comportamiento a la persona especifica.

ⓘ Ejemplo — Memoria tipo User
---
name: Kevin - Perfil
description: Desarrollador backend senior aprendiendo React, prefiere explicaciones concisas
type: user
---

Kevin es desarrollador backend con 5 anos de experiencia en Go.
Esta aprendiendo React y el frontend de este proyecto.
Prefiere explicaciones concisas con analogias al backend.

Cuando guardar: Al aprender detalles del rol, preferencias o conocimiento del usuario.

Como usar: Adaptar respuestas al perfil. Un senior no necesita que le expliques conceptos basicos.

2. Feedback (retroalimentacion)

Guia sobre como trabajar — que evitar Y que seguir haciendo. Es critico registrar tanto fracasos como exitos. Si solo guardas correcciones, Claude evitara errores pasados pero se alejara de enfoques que ya validaste.

ⓘ Ejemplo — Memoria tipo Feedback
---
name: No mockear la base de datos en tests
description: Tests de integracion deben usar DB real, no mocks, por incidente Q1
type: feedback
---

Tests de integracion deben usar una base de datos real, no mocks.

**Why:** Incidente en Q1 donde tests con mocks pasaron pero la
migracion de prod fallo porque el schema real era diferente.

**How to apply:** En cualquier test bajo src/tests/integration/,
siempre usar testcontainers con Postgres real.

Cuando guardar: Correcciones ("no hagas eso") O confirmaciones ("si, exactamente asi").

✔ Registrar exitos, no solo fracasos
---
name: PR bundleado para refactors
description: Para refactors, usuario prefiere 1 PR grande en vez de muchos pequenos
type: feedback
---

Para refactors en este area, el usuario prefiere un PR bundleado
en vez de muchos PRs pequenos.

**Why:** Confirmado despues de que elegi este approach y el usuario
lo aprobo sin objeciones. Separarlo habria sido churn innecesario.

**How to apply:** En refactors que tocan multiples archivos
relacionados, proponer un solo PR.

3. Project (proyecto)

Informacion sobre trabajo en curso, metas, bugs o incidentes que no se pueden derivar del codigo o git history. Estas memorias decaen rapido, asi que el why ayuda a juzgar si la memoria sigue siendo relevante.

ⓘ Ejemplo — Memoria tipo Project
---
name: Migracion REST a GraphQL
description: Migracion en curso de REST a GraphQL, deadline Q3 2026
type: project
---

Estamos migrando de REST a GraphQL. Deadline: Q3 2026.

**Why:** El equipo mobile necesita queries flexibles y el
REST actual genera over-fetching significativo.

**How to apply:** Nuevos endpoints deben ser GraphQL.
Los REST existentes se mantienen hasta migracion completa.
No agregar nuevos endpoints REST.

Cuando guardar: Al aprender quien hace que, por que o para cuando. Siempre convertir fechas relativas a absolutas ("el jueves" → "2026-04-03").

4. Reference (referencia)

Punteros a donde encontrar informacion en sistemas externos. No el contenido en si, sino la ubicacion.

ⓘ Ejemplo — Memoria tipo Reference
---
name: Dashboard de latencia API
description: Grafana dashboard que oncall monitorea para latencia de requests
type: reference
---

El dashboard de latencia API esta en grafana.internal/d/api-latency.
Es lo que oncall monitorea. Revisarlo antes de tocar codigo
de request handling.

Cuando guardar: Al aprender sobre recursos en sistemas externos (Linear, Slack, Grafana, wikis, etc.).

Resumen de los 4 tipos

Tipo Que guarda Ejemplo de trigger
user Rol, preferencias, conocimiento del usuario "Soy data scientist investigando el logging"
feedback Como trabajar: que evitar y que repetir "No, no uses mocks ahi" o "Si, perfecto asi"
project Trabajo en curso, metas, deadlines, contexto "Estamos en merge freeze hasta el viernes"
reference Donde encontrar info externa "Los bugs se trackean en Linear proyecto INGEST"

4.3 — Estructura de archivos

MEMORY.md — El indice

MEMORY.md es un indice, no una memoria. Cada entrada es 1 linea con menos de ~150 caracteres. Sin frontmatter. 

# Ejemplo de MEMORY.md bien estructurado

- [Kevin - Perfil](user_profile.md) — Backend senior aprendiendo React, prefiere conciso
- [No mockear DB](feedback_testing.md) — Tests integracion con DB real por incidente Q1
- [PR bundleado](feedback_pr_style.md) — Prefiere 1 PR grande para refactors
- [Migracion GraphQL](project_migration.md) — REST a GraphQL, deadline Q3 2026
- [Dashboard latencia](reference_grafana.md) — grafana.internal/d/api-latency para oncall
- [Bugs en Linear](reference_linear.md) — Proyecto INGEST para bugs de pipeline

Archivos de memoria individuales

Cada memoria es un archivo .md con frontmatter YAML obligatorio:

---
name: {nombre de la memoria}
description: {1 linea — se usa para decidir relevancia en futuras sesiones}
type: {user | feedback | project | reference}
---

{contenido de la memoria}

# Para feedback y project, usar esta estructura:
{regla o hecho}

**Why:** {motivacion, incidente o preferencia que lo origino}
**How to apply:** {cuando y donde aplicar esta regla}

Proceso de guardar memoria (2 pasos)

Paso Accion Herramienta
1 Escribir el archivo de memoria con frontmatter Write tool
2 Agregar puntero de 1 linea en MEMORY.md Edit tool

4.4 — Que NO guardar en memoria

No guardar Por que Alternativa
Patrones de codigo, arquitectura, estructura de archivos Se puede derivar leyendo el codebase actual Claude lo lee cuando lo necesita
Git history, cambios recientes, quien cambio que git log / git blame son la fuente de verdad Consultar git directamente
Soluciones de debugging ya aplicadas El fix esta en el codigo; el commit message tiene el contexto Revisar commits
Lo que ya esta en CLAUDE.md Duplicacion innecesaria Mantener en CLAUDE.md
Detalles efimeros de tarea actual Solo util en esta conversacion Usar Tasks o Plans
Listas de PRs o resumenes de actividad Se vuelven stale inmediatamente Preguntar que fue sorprendente y guardar eso
✔ Esto SI vale guardar
# No-obvio, recurrente, estable:
"El CI tarda 45 min porque corre E2E contra 3 browsers.
Para PRs que solo tocan backend, se puede skipear con [skip-e2e]."

# Preferencia validada:
"Kevin prefiere que le muestre el diff completo, no un resumen."

# Contexto no derivable del codigo:
"El modulo billing esta congelado por revision legal hasta Q3."
✘ Esto NO vale guardar
# Derivable del codigo:
"El proyecto usa TypeScript con ESLint y Prettier."
# Claude lo ve al leer package.json y tsconfig.json

# Git history:
"Kevin hizo 12 PRs esta semana, 3 en el modulo auth."
# git log lo dice mejor y mas actualizado

# Efimero:
"Estamos en el paso 3 de 7 del refactor actual."
# Usar Tasks para esto, no memoria

4.5 — Memoria vs otras formas de persistencia

Mecanismo Usar para Scope Quien lo escribe
Auto Memory Info util en futuras conversaciones Entre conversaciones Claude
CLAUDE.md Instrucciones permanentes que quieres forzar Entre conversaciones Tu
Plans Alinear approach de implementacion antes de empezar Conversacion actual Claude + tu
Tasks Trackear pasos discretos y progreso Conversacion actual Claude
⚠ Regla simple para decidir

Necesito esto en la proxima conversacion? → Memory o CLAUDE.md
Necesito esto solo ahora? → Plans o Tasks
Quiero que Claude lo siga siempre? → CLAUDE.md
Es un aprendizaje que Claude descubrio? → Auto Memory

4.6 — Comandos y settings

Comando /memory

  • Lista todos los CLAUDE.md y rules cargados en la sesion
  • Toggle auto memory on/off
  • Link para abrir la carpeta de memoria en tu editor
  • Seleccionar un archivo para editarlo directamente

Settings

// Desactivar auto memory
{ "autoMemoryEnabled": false }

// Directorio custom de memoria
{ "autoMemoryDirectory": "~/mi-memoria-custom" }
⚠ Seguridad

autoMemoryDirectory no se acepta desde .claude/settings.json (proyecto compartido) para prevenir que un repo rediriga escrituras de memoria a ubicaciones sensibles. Solo funciona desde policy, local o user settings.

Variable de entorno para CI/pipelines

# Desactivar completamente en CI (toma precedencia sobre todo)
export CLAUDE_CODE_DISABLE_AUTO_MEMORY=1

Guardar y borrar manualmente

Accion Como
Guardar memoria Di "recuerda que usamos pnpm, no npm"
Agregar a CLAUDE.md en vez de memoria Di "agrega esto al CLAUDE.md"
Editar memorias /memory → seleccionar archivo, o editar directamente en el filesystem
Borrar memorias Di "olvida lo de pnpm" o borra el archivo manualmente

Verificacion de memorias stale

Claude debe verificar memorias antes de actuar sobre ellas:

Si la memoria nombra... Verificar con...
Un archivo Confirmar que el archivo existe
Una funcion o flag Grep para confirmar que sigue ahi
Estado del repo Preferir git log sobre la memoria
⚠ Regla critica

"La memoria dice que X existe" no es lo mismo que "X existe ahora." Las memorias pueden quedar desactualizadas despues de refactors, renames o eliminaciones. Siempre verificar antes de recomendar basado en memoria.

4.7 — Las 10 mejores practicas

1. Revisar periodicamente

Usa /memory para auditar. Borra entries stale despues de refactors grandes.

2. Organizar por tema

Semantico, no cronologico. Agrupa por area (testing, deploy, preferencias).

3. Merge, no duplicar

Actualiza archivos existentes en vez de crear memorias casi identicas.

4. Fechas absolutas

Convertir "el jueves" a "2026-04-03" para que la memoria tenga sentido despues.

5. MEMORY.md < 200 lineas

Lo que excede no se carga al inicio. Mover detalle a topic files.

6. Entries < 150 chars

Cada linea de MEMORY.md debe ser un puntero conciso, no contenido.

7. Detalle en topic files

El contenido real va en archivos individuales, no en el indice.

8. Desactivar en CI

CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 para pipelines automatizados.

9. Feedback de exitos Y fracasos

No solo correcciones. Registrar tambien cuando un approach funciono bien.

10. Verificar antes de recomendar

Las memorias pueden estar desactualizadas. Grep/Read antes de actuar.
