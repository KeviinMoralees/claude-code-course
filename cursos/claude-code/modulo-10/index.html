10.1 — Prompt Engineering para Claude Code

La restricción fundamental de Claude Code es el context window: se llena rápido y el rendimiento degrada al llenarse. Toda técnica efectiva de prompt engineering existe para gestionar esta restricción.

La restricción que lo explica todo

El context window contiene toda la conversación: cada mensaje, cada archivo que Claude lee, cada output de comando. Una sola sesión de debugging puede consumir decenas de miles de tokens. Cuando el contexto se llena, Claude puede empezar a "olvidar" instrucciones anteriores o cometer más errores.

El ciclo de 4 fases

Separar la exploración de la implementación evita solucionar el problema equivocado:

1. Explorar

Entrar en Plan Mode. Claude lee archivos y responde preguntas sin hacer cambios.

2. Planear

Pedir un plan detallado. Ctrl+G abre el plan en tu editor para editarlo antes de continuar.

3. Implementar

Volver a Normal Mode. Claude ejecuta el plan verificando contra los criterios que definiste.

4. Commitear

Claude hace commit con mensaje descriptivo, push y abre el PR.

# Fase 1 — Explorar (Plan Mode)
read /src/auth and understand how we handle sessions.
also look at how we manage environment variables for secrets.

# Fase 2 — Planear
I want to add Google OAuth. What files need to change?
What's the session flow? Create a detailed plan.

# Fase 3 — Implementar (Normal Mode)
implement the OAuth flow from your plan.
write tests for the callback handler, run the suite and fix failures.

# Fase 4 — Commitear
commit with a descriptive message and open a PR
ℹ Cuándo saltarse las 4 fases

El ciclo agrega overhead. Para tareas pequeñas y claras ("corrige el typo en línea 42", "renombra esta variable") ve directo a implementar. El planning es más valioso cuanto más archivos toca el cambio o más incertidumbre hay sobre el enfoque.

CLAUDE.md efectivo — la regla de oro

Por cada línea, pregunta: "¿Causaría errores en Claude eliminarla?" Si no, elimínala. Un CLAUDE.md demasiado largo hace que Claude ignore las instrucciones importantes porque se pierden en el ruido.

Entrevistar a Claude antes de implementar

Para features grandes, deja que Claude haga las preguntas difíciles primero:

I want to build [brief description]. Interview me using AskUserQuestion.

Ask about technical implementation, UI/UX, edge cases, concerns, and tradeoffs.
Don't ask obvious questions — dig into the hard parts I might not have considered.

Keep interviewing until we've covered everything, then write a complete spec to SPEC.md.

Una vez generada la spec, iniciar una sesión nueva para implementar — contexto limpio, enfocado en la implementación.

✓ Dar a Claude un mecanismo de verificación

Es la técnica de mayor impacto. Sin criterios de éxito claros, Claude produce algo que "se ve bien" pero no funciona, y tú eres el único feedback loop. 

# Mal — sin verificación
implement a function that validates email addresses

# Bien — con criterios explícitos
write a validateEmail function.
test cases: user@example.com → true, invalid → false, user@.com → false
run the tests after implementing and fix any failures.
✗ Patrones de fallo comunes
  • La sesión kitchen sink: empiezas con una tarea, agregas otra no relacionada, vuelves a la primera. Contexto lleno de ruido. Solución: /clear entre tareas.
  • Corregir repetidamente: Claude hace algo mal, corregís, sigue mal, corregís de nuevo. Contexto contaminado con intentos fallidos. Solución: después de 2 correcciones, /clear y escribir un prompt mejor.
  • La exploración infinita: pides "investigar" algo sin scope. Claude lee cientos de archivos. Solución: scope la investigación o usa subagentes.

10.2 — Multi-Agent Workflows

Claude Code escala horizontalmente. Una vez que dominas una sesión, puedes multiplicar tu output con sesiones paralelas, modo no-interactivo y patrones de fan-out.

Patrón Writer/Reviewer

Un contexto fresco mejora la revisión de código — Claude no tiene bias hacia código que él mismo escribió. Usar dos sesiones separadas:

Session A (Writer)Session B (Reviewer)
Implement a rate limiter for our API endpoints
Review @src/middleware/rateLimiter.ts for edge cases, race conditions, and consistency with existing patterns.
Here's the review: [output]. Address these issues.

Patrón TDD con sesiones separadas

# Session A — solo escribe tests
Write comprehensive tests for a UserAuthService class.
Define the interface and all edge cases, but don't implement yet.

# Session B — implementa para pasar los tests
Implement UserAuthService to pass all tests in @src/auth/userAuth.test.ts
Run the tests after each change and fix failures.

Fan-out para operaciones masivas

# Paso 1: Claude genera la lista de archivos a migrar
list all Python files that use the old requests library (output: files.txt)

# Paso 2: Script itera con una invocación por archivo
for file in $(cat files.txt); do
  claude -p "Migrate $file from requests to httpx.
    Keep the same logic. Return OK or FAIL." \
    --allowedTools "Edit,Bash(git commit *)" &
done
wait

# Paso 3: Testear en 2-3 archivos primero, refinar el prompt, luego full run

Modo no-interactivo — integración con scripts y CI

# Output de texto plano (default)
claude -p "Explain what this project does"

# JSON estructurado para parseo
claude -p "List all API endpoints" --output-format json

# Streaming JSON en tiempo real
claude -p "Analyze this log file" --output-format stream-json

# Pipe a otro comando
claude -p "<prompt>" --output-format json | jq '.result'

# Como linter en scripts de CI
claude -p 'Review the diff vs main for security issues.
  Report: filename:line — description. No other text.'

Subagentes para mantener el contexto limpio

Los subagentes corren en su propio context window y reportan un resumen. La exploración de archivos no contamina tu conversación principal. 

# Delegar investigación
Use subagents to investigate how our auth system handles token refresh
and whether we have any existing OAuth utilities to reuse.

# Delegar verificación post-implementación
Use a subagent to review this code for edge cases and security issues.

# Configurar subagente especializado
# .claude/agents/security-reviewer.md
---
name: security-reviewer
description: Reviews code for security vulnerabilities. Use for any security audit.
tools: Read, Grep, Glob, Bash
model: opus
---
You are a senior security engineer. Review code for:
- SQL injection, XSS, command injection
- Authentication and authorization flaws
- Hardcoded secrets or credentials
Provide specific line references and suggested fixes.

Agent Teams (experimental)

Múltiples sesiones completas de Claude coordinadas por un team lead mediante mensajería. 

# Habilitar en settings.json o como variable de entorno
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

⚠ Costos de Agent Teams

Cada teammate mantiene su propio context window como instancia separada de Claude. Los costos son aproximadamente proporcionales al número de teammates activos. Usar Sonnet para teammates, equipos pequeños, y prompts de spawn enfocados. Terminar equipos cuando terminan su trabajo — teammates activos siguen consumiendo tokens aunque estén idle.

10.3 — Debugging con Claude

El debugging efectivo con Claude requiere darle la información correcta desde el inicio y los medios para verificar que el fix funciona.

El principio central: verificación propia

Si Claude no puede verificar que su fix funciona, puede producir algo que "se ve bien" pero no resuelve el problema. Siempre incluir un mecanismo de verificación: 

# Sin verificación — Claude no puede saber si funcionó
fix the login bug

# Con verificación — Claude puede iterar hasta que pase
Users report login fails after session timeout.
Check the auth flow in src/auth/, especially token refresh.
Write a failing test that reproduces the issue, then fix it.
The test must pass before you're done.

Dar la información del error correcta

# Dar el error exacto, no una descripción
The build fails with this error:
[paste exact error output]

Fix it and verify the build succeeds.
Address the root cause — don't suppress the error.

# Para errores intermitentes
Login fails ~20% of the time with: [error]
It's intermittent — likely a race condition or session timing issue.
Check the auth flow in src/auth/ and write a test that reliably reproduces it.

# Con contexto de reproducción
Steps to reproduce:
1. Login with valid credentials
2. Wait 30 minutes (session timeout)
3. Try to access /dashboard
4. Error: 401 Unauthorized

Check src/auth/session.ts and src/middleware/auth.ts

Debugging con Plan Mode

claude --permission-mode plan

Analyze the authentication system.

# (Shift+Tab para salir de Plan Mode)
Now implement the fix for the race condition you identified.

Patrones de debugging efectivos

# Debugging de performance — dar contexto de profiling
The API endpoint /users takes 800ms. Here's the flame graph: @profile.png
Identify the bottleneck and fix it. Target: under 200ms.

# Debugging de tests flaky
This test fails ~30% of the time: @src/__tests__/userService.test.ts
Failures always show: "Timeout waiting for database"
The test doesn't mock the DB. Identify the root cause and fix it.

# Debugging con subagente para no contaminar contexto
Use a subagent to investigate why the payment flow fails for users
with special characters in their email. Check src/payments/ thoroughly.
✓ Loop de feedback apretado

Corregir a Claude tan pronto como notes que va por mal camino. Esc detiene a Claude inmediatamente (el contexto se conserva). Redirigir es más eficiente que dejar que termine y luego deshacer. Si corregiste el mismo problema más de 2 veces: /clear y reescribir el prompt inicial con lo que aprendiste.

⚠ Medium es el default correcto

Usar high o max en tareas rutinarias puede hacer que Claude "piense demasiado" y sobre-complique soluciones simples. Reservar niveles altos para problemas que genuinamente lo necesitan.

Prompt caching (automático)

Claude Code usa prompt caching por defecto — el system prompt, CLAUDE.md y otros contenidos repetidos se reutilizan entre requests, reduciendo latencia y costo. 

# Deshabilitar caching para todos los modelos
export DISABLE_PROMPT_CACHING=1

# Solo para un modelo específico
export DISABLE_PROMPT_CACHING_SONNET=1
export DISABLE_PROMPT_CACHING_OPUS=1
export DISABLE_PROMPT_CACHING_HAIKU=1

10.5 — Seguridad

Claude Code usa una arquitectura basada en permisos con protecciones integradas, pero la seguridad final depende de las decisiones del usuario.

Protecciones integradas

Restricción de escritura

Claude solo puede escribir en el directorio donde se inició y sus subdirectorios. No puede modificar directorios padre sin permiso explícito.

Blocklist de comandos

curl y wget bloqueados por default — pueden traer contenido malicioso del exterior.

Contexto separado para web fetch

El fetch de URLs usa un context window separado para evitar inyectar prompts maliciosos en la conversación principal.

Verificación de fuentes nuevas

Primera ejecución en un codebase y nuevos servidores MCP requieren verificación de confianza explícita.

Prompt injection — mitigaciones

Un atacante puede intentar insertar instrucciones maliciosas en archivos que Claude lee. Las protecciones actuales:

  • Análisis de contexto completo: detecta instrucciones potencialmente dañinas analizando el request completo
  • Comandos bash sospechosos: requieren aprobación manual incluso si están en la allowlist
  • Fail-closed matching: comandos no reconocidos requieren aprobación manual por default
  • Descripciones en lenguaje natural: comandos bash complejos incluyen explicación antes de pedir aprobación 
# Contenido que puede contener prompt injection — revisar con cuidado
cat external-config.yaml | claude -p "parse this"   # riesgo: config maliciosa
claude -p "review @untrusted-file.js"               # riesgo: archivo con instrucciones

# Mejor práctica: revisar el contenido antes de pasarlo a Claude

Modos de permiso para reducir riesgo

# Plan Mode — Claude solo lee, no modifica
claude --permission-mode plan

# Auto mode — clasificador revisa comandos automáticamente
# Bloquea: escalación de scope, infraestructura desconocida, contenido hostil
claude --permission-mode auto

# Sandbox — aislamiento a nivel OS (filesystem + network)
/sandbox    # desde dentro de Claude Code

# bypassPermissions — SOLO para sandboxes sin internet
# Habilitar en settings.json solo si sabes lo que haces
{ "dangerouslySkipPermissions": true }

✓ Mejores prácticas
  • Revisar todos los cambios propuestos antes de aprobar — especialmente scripts bash con efectos secundarios
  • Usar devcontainers para aislamiento adicional en proyectos sensibles
  • Auditar configuración periódicamente con /permissions
  • Usar VS Code Restricted Mode cuando trabajas con código no confiable
  • Para equipos: usar managed-settings.json para forzar políticas organizacionales
  • Monitorear uso con OpenTelemetry si tienes equipos grandes
  • Reportar comportamiento sospechoso con /feedback
✗ Anti-patrones de seguridad
  • Pipear contenido no confiable a Claude: cat suspicious.txt | claude
  • Habilitar bypassPermissions fuera de un sandbox sin internet
  • Dar permisos de escritura irrestrictos en repos con código sensible
  • Confiar ciegamente en MCP servers de terceros no auditados
  • Habilitar WebDAV en Windows — riesgo de network requests bypass
  • No revisar cambios a archivos de configuración críticos (migrations, CI, secrets)
