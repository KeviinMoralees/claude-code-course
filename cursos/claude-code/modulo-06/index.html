Skills

Capacidades reutilizables que extienden a Claude: instrucciones, codigo y recursos cargados bajo demanda

6.1 — Commands vs Skills: diferencia clave

Son dos conceptos distintos. Confundirlos lleva a errores de configuracion y expectativas incorrectas.

Commands built-in

Son comandos hardcoded en el binario de Claude Code. No los creas, no los modificas, no viven en archivos. Ejecutan logica fija del sistema cuando los invocas con /.

ComandoFuncion
/helpMuestra ayuda general de Claude Code
/compact [focus]Compacta el contexto con instruccion de foco opcional
/clearLimpia la conversacion completa
/contextMuestra el uso actual del contexto por categoria
/mcpLista y administra servidores MCP activos
/permissionsVer y editar reglas de permisos de la sesion
/initGenera un CLAUDE.md para el proyecto actual
/costMuestra el costo de tokens de la sesion
/doctorDiagnostico de instalacion y configuracion
/fastAlterna el modo rapido (Fast mode)
/memoryVer y editar archivos de memoria del sistema
/skillsListar skills disponibles en la sesion actual
Regla de oro
Los Commands built-in son de Claude Code, no tuyos. Las Skills son tuyas: las creas, las configuras y Claude las usa automaticamente o cuando las invocas.

Skills — el sistema moderno

Las Skills son archivos en disco que empaquetan instrucciones, recursos y codigo ejecutable. Claude las descubre y carga bajo demanda segun relevancia. Son el reemplazo y evolucion de los "custom commands" legacy (.claude/commands/).

Custom Commands (legacy)
El sistema antiguo usaba .claude/commands/deploy.md para crear /deploy. Sigue funcionando, pero Skills es el sistema unificado que lo reemplaza con mucho mas poder. Un archivo en .claude/commands/deploy.md y una skill en .claude/skills/deploy/SKILL.md producen el mismo /deploy — pero la skill admite frontmatter, archivos de soporte, scripts y control de invocacion.

6.2 — Arquitectura de una Skill

Estructura de directorios

~/.claude/skills/
└── mi-skill/
    ├── SKILL.md          ← Punto de entrada (obligatorio)
    ├── GUIDE.md          ← Guia adicional (opcional)
    ├── examples.md       ← Ejemplos concretos (opcional)
    ├── reference.md      ← Referencia detallada (opcional)
    └── scripts/
        ├── validate.py   ← Script ejecutable
        └── process.sh    ← Script ejecutable

Anatomia de SKILL.md

---
name: explain-code
description: Explains code with diagrams and analogies. Use when explaining
  how code works, teaching about a codebase, or when the user asks
  "how does this work?"
---

## When explaining code, always include:

1. **An analogy**: Compare the code to something from everyday life
2. **A diagram**: Use ASCII art to show flow or relationships
3. **Step-by-step walkthrough**: What happens at each stage
4. **A gotcha**: A common mistake or misconception

Keep explanations conversational. For complex concepts, use multiple analogies.

Los tres niveles de carga (Progressive Disclosure)

NivelCuando cargaCosto tokensContenido
1 — MetadataSiempre, al inicio~100 tokens por skillname y description del YAML
2 — InstruccionesCuando la skill es relevante<5k tokensCuerpo completo de SKILL.md
3 — RecursosSolo si se necesitanSin limite practicoArchivos extra, scripts (solo output al ejecutar)
Por que importa la carga progresiva
Puedes tener decenas de skills instaladas sin penalizar el contexto. Solo la metadata (nombre + descripcion) de todas las skills vive en el sistema prompt. El cuerpo de SKILL.md solo se carga cuando Claude decide que esa skill es relevante para tu tarea. Los scripts nunca entran al contexto — solo su output.

6.3 — Donde viven las Skills

AlcanceRutaSe aplica a
EnterpriseManaged settings (admin)Todos los usuarios de la organizacion
Personal~/.claude/skills/<skill>/SKILL.mdTodos tus proyectos
Proyecto.claude/skills/<skill>/SKILL.mdSolo este proyecto
Plugin<plugin>/skills/<skill>/SKILL.mdDonde el plugin este activo

Cuando skills comparten el mismo nombre en distintos niveles, gana la mayor precedencia: enterprise > personal > proyecto > plugin.

Monorepos
Claude Code descubre skills automaticamente en subdirectorios. Si tienes packages/api/.claude/skills/, las skills ahi se activan cuando trabajas con archivos de ese subdirectorio.

6.4 — Skills bundled (incluidas en Claude Code)

Estas skills vienen preinstaladas y disponibles en todas las sesiones. A diferencia de los commands built-in, estan implementadas como skills de prompt: le dan a Claude un playbook detallado y dejan que orqueste el trabajo con sus herramientas.

SkillProposito
/batch <instruccion> Cambios masivos en paralelo. Descompone el trabajo en 5-30 unidades independientes, crea un plan, luego lanza un agente por unidad en worktrees aislados. Cada agente implementa, corre tests y abre un PR. Requiere repositorio git.
/claude-api Carga referencia de la API de Claude para el lenguaje del proyecto (Python, TS, Go, Java, Ruby, C#, PHP, cURL). Se activa automaticamente cuando detecta imports de anthropic o @anthropic-ai/sdk.
/debug [descripcion] Activa debug logging para la sesion actual y analiza el log para diagnosticar problemas. El debug esta desactivado por default a menos que hayas iniciado con claude --debug.
/loop [intervalo] <prompt> Ejecuta un prompt repetidamente mientras la sesion este abierta. Util para polling de deploys, monitoreo de PRs, tareas periodicas. Ej: /loop 5m check if the deploy finished.
/simplify [foco] Lanza 3 agentes de revision en paralelo sobre los archivos recien modificados, agrega sus hallazgos y aplica correcciones de reutilizacion, calidad y eficiencia. Ej: /simplify focus on memory efficiency.

6.5 — Frontmatter: todos los campos

---
name: deploy                   # max 64 chars, solo lowercase/numeros/guiones
description: "Deploy to prod"  # max 1024 chars, tercera persona, sin XML
argument-hint: "[env]"         # hint en autocompletado

disable-model-invocation: true  # solo TU lo invocas, Claude no
user-invocable: false          # solo Claude lo usa, no aparece en /menu

context: fork                  # ejecuta en subagente aislado
agent: Explore                 # tipo de subagente (Explore, Plan, general-purpose)

allowed-tools: Read, Grep, Bash(git *)  # limitar herramientas disponibles
model: claude-opus-4-6        # modelo especifico para esta skill
effort: high                   # low | medium | high | max
paths: "src/**/*.ts"          # activar solo para archivos que coincidan
shell: bash                    # shell para bloques !`command` (bash o powershell)
---

Variables disponibles

6.7 — Contexto dinamico con !`command`

La sintaxis !`comando` ejecuta comandos shell antes de que Claude vea el contenido de la skill. El output reemplaza el placeholder. Esto es preprocesamiento, no algo que Claude ejecuta. 

---
name: pr-summary
description: Summarizes changes in the current pull request
context: fork
agent: Explore
allowed-tools: Bash(gh *)
---

## Pull request context
- Diff: !`gh pr diff`
- Comments: !`gh pr view --comments`
- Files changed: !`gh pr diff --name-only`
- Current branch: !`git branch --show-current`

## Your task
Write a clear, concise PR summary covering:
1. What was changed and why
2. Key technical decisions
3. Testing approach
Extended thinking en skills
Para activar razonamiento extendido (ultrathink) dentro de una skill, incluye la palabra ultrathink en cualquier parte del contenido de SKILL.md.

6.8 — Skills en subagentes (context: fork)

Con context: fork, el contenido de SKILL.md se convierte en el prompt que maneja un subagente aislado. El subagente no tiene acceso al historial de la conversacion. 

---
name: deep-research
description: Researches a topic thoroughly across the codebase
context: fork
agent: Explore
---

Research $ARGUMENTS thoroughly:

1. Find relevant files using Glob and Grep
2. Read and analyze the code
3. Identify patterns, dependencies, edge cases
4. Summarize findings with specific file:line references
EnfoqueSystem promptTaskTambien carga
Skill con context: forkDel tipo de agente (Explore, Plan...)Contenido de SKILL.mdCLAUDE.md
Subagente con campo skillsMarkdown del subagenteMensaje de delegacion de ClaudeSkills precargadas + CLAUDE.md
Advertencia
context: fork solo tiene sentido para skills con instrucciones de tarea explicitas. Si tu skill solo contiene guias o conocimiento de fondo sin una tarea concreta, el subagente retornara sin hacer nada util.

6.9 — Buenas practicas (documentacion oficial)

Principio 1: Conciso es clave

El contexto es un bien compartido. Tu skill compite por tokens con el historial de conversacion, otras skills y la solicitud actual. Solo agrega lo que Claude no sabe ya.

Bien — conciso (~50 tokens)
## Extract PDF text

Use pdfplumber for text extraction:

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```
Mal — verboso (~150 tokens)
## Extract PDF text

PDF (Portable Document Format) files are a common file format that contains
text, images, and other content. To extract text from a PDF, you'll need to
use a library. There are many libraries available for PDF processing, but
pdfplumber is recommended because it's easy to use and handles most cases well.
First, you'll need to install it using pip. Then you can use the code below...

Claude ya sabe que es un PDF y como funcionan las librerias. No lo expliques.

Principio 2: Nivel de libertad apropiado

Ajusta la especificidad de las instrucciones segun la fragilidad de la tarea.

Alta libertad (texto)

Multiples enfoques son validos, las decisiones dependen del contexto. Ej: code review, analisis.

Media libertad (pseudocodigo)

Hay un patron preferido pero con variacion aceptable. Ej: generacion de reportes con parametros.

Baja libertad (script exacto)

La operacion es fragil, la consistencia es critica. Ej: migraciones de DB que deben seguir un orden exacto.

Principio 3: Nombres y descripciones efectivas

Bien — descripcion especifica
description: Extract text and tables from PDF files, fill forms, merge documents.
  Use when working with PDF files or when the user mentions PDFs, forms,
  or document extraction.
Mal — descripcion vaga
description: Helps with documents

Claude no puede saber cuando usar esta skill vs otra que tambien "ayuda con documentos".

Reglas para nombres: usa gerundio (processing-pdfs, analyzing-spreadsheets, managing-databases). Evita nombres vagos como helper, utils, tools.

Reglas para descripciones: escribe en tercera persona ("Processes Excel files" — no "I can help you"). Incluye tanto que hace la skill como cuando usarla. Las descripciones se truncan a 250 caracteres si hay muchas skills instaladas, asi que pon lo mas importante primero.

Principio 4: Divulgacion progresiva

Bien — SKILL.md corto con referencias
## Quick start
Run `analyze_form.py` to inspect a PDF form.

## Additional resources
- For form filling, see [FORMS.md](FORMS.md)
- For API reference, see [reference.md](reference.md)
Mal — todo en SKILL.md
Meter 800 lineas de documentacion, ejemplos, referencia de API y casos edge en un solo SKILL.md. Esto carga todo en contexto aunque solo se necesite el 10%.

Mantener SKILL.md bajo 500 lineas. Las referencias deben ser maximo 1 nivel de profundidad desde SKILL.md.

Principio 5: Feedback loops para calidad

## Workflow con verificacion

Task Progress:
- [ ] Step 1: Analyze the form (run analyze_form.py)
- [ ] Step 2: Create field mapping (edit fields.json)
- [ ] Step 3: Validate mapping (run validate_fields.py)
- [ ] Step 4: Fill the form (run fill_form.py)
- [ ] Step 5: Verify output (run verify_output.py)

Principio 6: Scripts que resuelven, no que delegan

Bien — el script maneja errores
def process_file(path):
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError:
        print(f"File {path} not found, creating default")
        with open(path, "w") as f:
            f.write("")
        return ""
    except PermissionError:
        print(f"Cannot access {path}, using default")
        return ""
Mal — dejar que el script falle a Claude
def process_file(path):
    with open(path) as f:    # Si el archivo no existe, explota
        return f.read()       # y Claude tiene que improvisar

Los errores no manejados obligan a Claude a improvisar recuperacion, gastando tokens y produciendo resultados inconsistentes.

Principio 7: Probar con todos los modelos

Las skills se comportan diferente segun el modelo que las usa. Prueba con los tres:

Claude Haiku

Rapido y economico. Pregunta: da la skill suficiente orientacion para que Haiku lo ejecute bien?

Claude Sonnet

Balanceado. Pregunta: es la skill clara y eficiente sin ser redundante?

Claude Opus

Razonamiento potente. Pregunta: evita la skill sobrexplicar cosas que Opus ya sabe?

Principio 8: Desarrollo iterativo con evaluaciones

Crea evaluaciones antes de escribir documentacion extensa. El proceso es:

  1. Ejecuta la tarea sin skill — identifica que falla
  2. Crea 3 evaluaciones que prueben esos gaps
  3. Mide el baseline de Claude sin skill
  4. Escribe el minimo de instrucciones para cubrir los gaps
  5. Itera: Claude A crea/refina la skill, Claude B la prueba en tareas reales

6.10 — Anti-patrones a evitar

Ofrecer multiples opciones sin default
# MAL
You can use pypdf, or pdfplumber, or PyMuPDF, or pdf2image, or...

# BIEN
Use pdfplumber for text extraction.
For scanned PDFs requiring OCR, use pdf2image with pytesseract instead.

Dale a Claude una respuesta, no una pregunta. Un default claro con una escotilla de escape para casos especiales.

Paths estilo Windows
# MAL
scriptshelper.py

# BIEN
scripts/helper.py

Siempre usa forward slashes, incluso en Windows.

Informacion que envejece
# MAL — quedara desactualizado
As of March 2025, the API supports...
In version 3.2.1, use the following...

# BIEN — atemporal
Use the latest stable version from npm.
Check the CHANGELOG.md for version-specific notes.
Constantes magicas sin explicacion
# MAL
REQUEST_TIMEOUT = 30
MAX_RETRIES = 3

# BIEN
# HTTP requests typically complete within 30 seconds
REQUEST_TIMEOUT = 30

# Three retries balances reliability vs speed
MAX_RETRIES = 3

6.11 — Checklist para una Skill efectiva

Calidad del contenido

☐ Descripcion especifica con keywords clave ☐ Descripcion incluye que hace Y cuando usarla ☐ SKILL.md bajo 500 lineas ☐ Detalles extra en archivos separados ☐ Sin informacion que envejece ☐ Terminologia consistente ☐ Referencias maximo 1 nivel de profundidad

Scripts y codigo

☐ Scripts manejan errores internamente ☐ Sin constantes magicas (todas justificadas) ☐ Paquetes requeridos listados y verificados ☐ Sin paths estilo Windows ☐ Pasos de validacion para operaciones criticas ☐ Feedback loops para tareas de calidad critica

Testing

☐ Al menos 3 evaluaciones creadas ☐ Probada con Haiku, Sonnet y Opus ☐ Probada con escenarios reales ☐ Feedback del equipo incorporado ☐ Iterada con Claude A (crea) y Claude B (prueba)
