1.1 — Que es Claude Code

Claude Code es la herramienta de codificacion agentica de Anthropic. Lee tu codebase, edita archivos, ejecuta comandos y se integra con tus herramientas de desarrollo. No es un simple chatbot: es un agente que puede planificar, ejecutar y verificar su propio trabajo.

6 plataformas, 1 mismo motor

VS Code / CursorExtension con diffs inline, @-mentions, revision de planesExtensions Marketplace → buscar "Claude Code"

ℹ Dato clave

Todas las superficies comparten el mismo motor. Tu CLAUDE.md, settings y MCP servers funcionan en todas las plataformas sin cambios.

Capacidades principales

Construir features

Describe lo que quieres en lenguaje natural. Claude planifica, escribe codigo en multiples archivos y verifica que funcione.

Automatizar tareas

Tests, lint, merge conflicts, actualizar dependencias, release notes — todo lo tedioso.

Operaciones Git

Stage, commits, branches, PRs. Todo desde la conversacion.

MCP Integration

Conecta a Google Drive, Jira, Slack, bases de datos y cualquier herramienta via Model Context Protocol.

Equipos de agentes

Lanza multiples agentes Claude trabajando en paralelo en diferentes partes de una tarea.

Composable con Unix

Pipe logs, ejecuta en CI, encadena con otras herramientas. Se comporta como una herramienta Unix mas.

1.2 — Modos de Ejecucion

Modo Interactivo (default)

Es el modo principal. Abre un TUI completo donde conversas con Claude.

# Abrir el TUI
claude

# Abrir con un prompt inicial
claude "explica la arquitectura de este proyecto"

# Reanudar la ultima sesion
claude -c

# Reanudar una sesion especifica
claude -r "session-id-o-nombre"

Modo Print / Headless (-p)

No interactivo. Ejecuta, imprime el resultado y sale. Ideal para scripts, CI/CD y automatizacion.

# Pregunta simple
claude -p "Que hace el modulo de auth?"

# Con salida JSON estructurada
claude -p "lista las dependencias" --output-format json

# Limitar gasto y turnos
claude -p "refactoriza el modulo X" --max-turns 10 --max-budget-usd 5

# Modo bare: sin hooks, MCP, CLAUDE.md (rapido y limpio para CI)
claude --bare -p "resume este archivo"
FlagQue hace
--output-format text|json|stream-jsonFormato de salida. json incluye session ID y metadata
--json-schema '{...}'Salida JSON validada contra un schema (structured outputs)
--allowedTools "Bash,Read"Auto-aprueba herramientas especificas
--max-turns NLimita turnos agenticos
--max-budget-usd NLimite de gasto en dolares
--bareSin hooks, skills, MCP, CLAUDE.md. Recomendado para CI
--continue / --resume <id>Continuar o reanudar una conversacion previa
--fallback-model sonnetModelo alternativo si el principal esta sobrecargado

Modo Pipe

Pipe contenido de otras herramientas hacia Claude Code. Se combina con -p.

# Analizar logs en busca de anomalias
tail -200 app.log | claude -p "hay anomalias aqui?"

# Review de seguridad sobre archivos cambiados
git diff main --name-only | claude -p "revisa estos archivos por vulnerabilidades"

# Traducir strings y crear PR
claude -p "traduce los strings nuevos al frances y crea un PR"
✔ Buena practica

Usa --bare en CI/CD y scripts. Arranca mas rapido y evita efectos secundarios de hooks o MCP servers que podrian no estar disponibles en ese entorno.

# En tu pipeline de CI
claude --bare -p "revisa este PR por problemas" --output-format json
✘ Mala practica

Usar modo interactivo en scripts automatizados. Se quedara esperando input del usuario y el script se colgara.

# MAL: esto se cuelga en CI porque espera input
claude "haz el deploy"

# BIEN: usa -p para que sea no-interactivo
claude -p "haz el deploy" --allowedTools "Bash,Read,Edit"

1.3 — Modelos Disponibles

Modelos y aliases

AliasModeloCuando usarlo
opusOpus 4.6Razonamiento complejo, tareas autonomas largas, arquitectura
sonnetSonnet 4.6Tareas diarias de codigo, balance rendimiento/costo
haikuHaiku 4.5Tareas simples, 4-5x mas rapido que Sonnet
sonnet[1m]Sonnet con 1M tokensSesiones largas con codebases grandes
opus[1m]Opus con 1M tokensSesiones largas con codebases grandes
opusplanOpus planifica → Sonnet ejecutaLo mejor de ambos: razonamiento de Opus + eficiencia de Sonnet

Como cambiar de modelo

# 1. Durante la sesion (slash command)
/model opus

# 2. Al iniciar Claude Code
claude --model opus

# 3. Variable de entorno
export ANTHROPIC_MODEL=opus

# 4. En settings.json
{ "model": "opus" }

Niveles de esfuerzo (Effort)

Controlan la profundidad del razonamiento. Mas esfuerzo = mas tokens = respuestas mas profundas.

NivelDescripcionCuando usarlo
lowRapido y baratoPreguntas simples, tareas triviales
mediumDefaultLa mayoria de tareas de codigo
highRazonamiento profundoDebugging complejo, decisiones de arquitectura
maxSin limite de tokens de razonamiento. Solo OpusProblemas extremadamente dificiles
# Cambiar esfuerzo en sesion
/effort high

# Al iniciar
claude --effort high

# Variable de entorno
export CLAUDE_CODE_EFFORT_LEVEL=high
✔ Buena practica

Usa opusplan para tareas grandes: Opus analiza y planifica la solucion, luego Sonnet ejecuta el codigo. Combinas la inteligencia de Opus con la velocidad de Sonnet.

✘ Mala practica

Usar opus con /effort max para tareas simples como renombrar una variable. Gastas tokens innecesariamente. Usa haiku o sonnet con /effort low para eso.

ℹ Sobre /fast

/fast usa el mismo modelo Opus 4.6 con salida acelerada. NO cambia a otro modelo. Es solo una optimizacion de velocidad de generacion.

1.4 — Sistema de Permisos

Principio fundamental: read-only por defecto

Claude Code no puede modificar nada sin tu permiso. Solo lectura es automatico; todo lo demas requiere aprobacion.

TipoEjemploRequiere permiso?
Solo lecturaRead, Grep, GlobNo
Comandos BashShell executionSi
ModificacionEdit, WriteSi

Modos de permisos

ModoQue puede hacer sin preguntarIdeal para
defaultSolo leer archivosEmpezar, trabajo sensible
acceptEditsLeer y editar archivosIterar en codigo que estas revisando
planSolo lectura (sin edits ni comandos)Explorar codebase, planificar refactors
autoTodo, con clasificador de seguridadTareas largas, reducir fatiga de prompts
bypassPermissionsTodo, sin checksSolo containers/VMs aislados
# Cambiar modo durante sesion
Shift+Tab    # Cicla: default → acceptEdits → plan → auto

# Al iniciar
claude --permission-mode plan

# En settings.json
{ "permissions": { "defaultMode": "acceptEdits" } }

Reglas allow / deny

Puedes pre-configurar que herramientas se aprueban o deniegan automaticamente. El orden de evaluacion es: deny → ask → allow (deny siempre gana).

{
  "permissions": {
    "allow": [
      "Bash(npm run *)",        // Cualquier script npm
      "Bash(git commit *)",     // Commits
      "Read",                    // Leer cualquier archivo
      "Edit(/src/**/*.ts)"      // Editar solo .ts en /src
    ],
    "deny": [
      "Bash(git push *)",       // Nunca push automatico
      "Read(./.env)"            // No leer secrets
    ]
  }
}
✔ Buena practica

Configura reglas deny para operaciones peligrosas y allow para las que usas frecuentemente. Asi reduces prompts sin comprometer seguridad.

// Seguro: permite leer y editar tu codigo, bloquea push y env
{
  "allow": ["Read", "Edit(/src/**)", "Bash(npm test *)"],
  "deny": ["Bash(git push *)", "Bash(rm -rf *)", "Read(.env*)"]
}
✘ Mala practica

Usar bypassPermissions en tu maquina local. Este modo no tiene ningun check de seguridad. Solo usalo en containers desechables.

// PELIGROSO: en tu maquina local esto podria borrar archivos importantes
claude --dangerously-skip-permissions -p "limpia el proyecto"

Auto Mode: el clasificador inteligente

Disponible en planes Team, Enterprise y API. Un modelo clasificador separado (Sonnet 4.6) revisa cada accion antes de ejecutarla. Bloquea automaticamente acciones peligrosas:

  • curl | bash — ejecutar scripts remotos
  • Enviar datos sensibles a servicios externos
  • Deploys a produccion
  • Borrado masivo, cambios IAM, force push
  • Push a main

Si el clasificador bloquea 3 acciones consecutivas o 20 en total, auto mode se pausa y vuelve a pedir permisos manuales.

Jerarquia de settings

De mayor a menor prioridad:

  1. Managed settings — Politica enterprise (no se puede sobreescribir)
  2. Argumentos CLI — Flags al ejecutar claude
  3. Settings locales — .claude/settings.local.json
  4. Settings compartidos — .claude/settings.json
  5. Settings de usuario — ~/.claude/settings.json
⚠ Recuerda

Las reglas denysiempre ganan sin importar donde esten definidas. Si un admin pone un deny en managed settings, ningun otro nivel puede sobreescribirlo.

Sandboxing

Aislamiento a nivel de sistema operativo que restringe acceso a filesystem y red del tool Bash. Es complementario al sistema de permisos: los permisos controlan que puede hacer Claude, el sandbox controla que puede acceder el shell. 

// Configuracion de sandbox en settings.json
{
  "sandbox": {
    "network": {
      "allowedDomains": ["registry.npmjs.org", "api.github.com"],
      "allowLocalBinding": false
    }
  }
}

# Activar sandbox durante sesion
/sandbox

Segun datos de Anthropic, sandboxing reduce los prompts de permisos en un ~84%.
